# 为什么为产品经理打造一个产品很难

![](img/c407d2a0aa05dd4afd500b9d4f7cd9a5.png)

2020 年 10 月，我们决定做一些具有挑战性的事情——为产品经理构建一个工具，以便在一个地方定义、管理和协作他们的产品。这将是一个项目经理将他们的想法变成成品所需要的最后一个工具。

有面向开发者的 GitHub、面向设计师的 Figma 和面向营销人员的 HubSpot。产品经理呢？

Zeda.io 就是从这种思想中诞生的，帮助集中产品数据，提高产品管理的效率。

# 一些历史

我们今天所知的产品管理始于 1931 年宝洁公司的尼尔·h·麦克罗伊写的一份备忘录。这只是雇佣更多员工的一个理由，但却成为了现代产品管理的基石。

如果我们必须用简单的语言来定义这个角色，那么产品管理是一项适合精通技术的人的工作，他们有着优秀的设计眼光和丰富的同情心。

但是请注意，产品经理的职责并没有固定的定义。它往往因行业和组织而异。

围绕产品管理的这种模糊性也使得很难为产品经理创建一个通用的工具来帮助他们更好地完成工作。

我们将通过列出原因来解释为什么。

# 1.产品管理不是一个定义明确的角色

由于项目经理的角色没有被很好地定义，所以很难将我们应该用这个产品解决的所有用例的变化都考虑进去。这不是一件容易的工作。为了确保我们没有遗漏任何人，我们特别注意了解他们的日常挑战和痛点。

到目前为止，我们已经和 500 多位产品经理谈过了，而且不打算停止。在我们的书里没有“足够的反馈”这样的东西。

# 2.不同的公司有所不同

产品管理也因公司而异。公司在制定产品战略时主要使用两种方法。一种是“自下而上”的方法，另一种是“自上而下”的方法。

微软和苹果使用自上而下的方法，而谷歌、脸书、LinkedIn 和亚马逊使用自下而上的方法。

具有透明文化的公司让每个团队都能看到其他团队在做什么。项目经理可能在不同的团队中从事项目工作，跨团队协作更加频繁。更加孤立的公司有专注于他们的项目的团队，并且尽量减少与其他团队的互动。

# 3.项目经理在不同的层次做不同的事情

早期项目经理的职责与更有经验的项目经理的职责有很大不同。这意味着产品经理不仅仅是一个目标群体——这里有基于资历级别的多个细分市场。这进一步增加了针对不同细分市场的特征的优先级排序的复杂性。

我们应该更关注早期经前综合症还是有经验的经前综合症？或者我们应该试着让每个人都开心？随着我们收集更多的用户反馈，这些都是我们仍在试图回答的问题。因此，如果您有任何反馈，请插话。:)

# 4.产品管理因阶段而异

在产品开发过程中，项目经理的工作变化很大。当产品处于早期阶段时，项目经理的角色主要是产品定义和策略，而在后期阶段，则是通过持续改进来保持产品的运转。

这意味着，如果我们想为经前综合症创造一个产品，它应该解决他们在所有阶段的挑战。

# 5.各行业差异很大

尽管 B2B 和 B2C 产品经理分享许多相同的角色，使用许多相同的技能和技术，但另一方面，他们的计划、优先级和实现会有很大的不同。

一些产品经理发现在这两种方法之间转换很容易。其他人可能会对其中一个有强烈的偏好。

另一个需要考虑的大问题是行业。产品管理在科技、电子商务、金融科技、数据科学等行业有很大不同。

当我们为产品经理构建工具时，我们必须确保它同样适用于产品管理的所有变化。这是我们正在用 [Zeda.io](https://zeda.io/) 解决的最大痛点之一。

# 6.有不同类型的经前综合症

产品经理可以分为 5 种主要类型，它们是:

**1。技术产品经理**

在转向产品之前，技术产品经理几乎总是开发人员。他们不仅仅是开发人员，他们还是你最好的开发人员之一。因此，技术项目经理知道什么可以快速发货，以及什么是值得投入时间的，以便有一个高效的产品发布。这种理解从他们做程序员的时候就延续下来了——使他们成为这个职位的唯一合格人选。

**2。分析/数据科学产品经理**

分析产品经理是技术产品经理(TPM)的近亲。这两个角色之间有重叠，但也有一些不同。在进入产品管理之前，TPM 几乎总是开发人员；而 APM 在加入之前也有数据分析或其他分析相关领域的经验。但不是任何一个分析师——他们必须是你最好的分析师之一。

**3。营销产品经理**

营销产品经理对最终客户有着与生俱来的理解。他们知道客户的目标，他们的角色，甚至是他们的动机。

通常，营销经理都有广告或公共关系方面的背景——可以说他们真的了解营销如何与客户合作，因为这是他们每天都在做的事情！对于任何一个产品经理来说，如果一个功能不仅能卖出去，还能赶走潜在的购买者，那么这个功能要么是在“卖”产品，要么根本不相关，这应该是显而易见的。

**4。“把事情做好”的产品经理**

“把事情做好”产品经理是标准产品经理的一个罕见但有效的变体。这位总理不接受否定的回答，并把事情做好。这种类型的产品经理通常只出现在高增长/高强度的初创公司和少数仍然充满活力的晚期私营公司。

**5。有远见的产品经理**

有远见的产品经理不像其他产品经理。他们不是整天呆在战壕里，而是对一家公司及其市场的方方面面有着与生俱来的理解。

这些人对他们的公司有更长远的眼光。他们不仅仅关注一个产品发布，而是关注成为比他们自身更大的事物的一部分意味着什么。他们的思维方式与传统企业环境中的思维方式非常不同——这些项目经理不考虑季度目标，而是考虑他们的工作如何影响一个组织的长期增长和成功。

# 7.他们戴许多帽子

产品管理是巨大的。对于那些不喜欢一辈子只做一件事的人来说，这是一份理想的工作。项目经理经常戴着开发人员、设计师、市场营销人员、经理的帽子，这样的例子不胜枚举。

项目经理不一定是最好的开发人员、设计人员或营销人员，但他们应该对这些学科有一个全面的了解，以发挥他们的最佳能力。有时候，任何其他团队不负责的事情都变成了产品经理的工作。这再次成为为 PMs 构建时要考虑的主要因素。

# 8.关于人员管理的更多信息

虽然人们普遍认为产品经理的工作只是定义和管理产品，但它在人员管理方面也很重要。这就是为什么所有伟大的产品经理都是杰出的人事经理。在构建 Zeda.io 时，我们必须对支持产品管理和人员管理的功能进行排序，以解决这个问题。

# 9.产品经理很难取悦

最后，产品经理知道什么是好的，什么是不好的。为产品经理打造产品的时候，逃不过他们挑剔的眼光。项目经理会很快发现漏洞、错误和坏的 UX。毫无疑问，他们也会有许多功能需求。这不一定是一件坏事，它只是提高了我们产品的标准，我们喜欢这样。

# 10.他们已经在使用一种临时解决方案

产品经理有本事找到解决他们可能面临的任何问题的方法。虽然市场上没有完全满足产品经理需求的专用工具，但智能项目经理已经在使用一些临时解决方案。这可以是使用 Zapier 集成连接的多个工具的形式，有时甚至是他们自己的定制编码工具。

然而，一个健壮的解决方案将使 PMs 的事情变得容易得多，这就是我们存在的原因。

如果你是一个产品经理，只要知道我们正在开发一个产品，它考虑了产品管理的所有上述细微差别，同时满足 PMs 众所周知的极端质量期望。

我们仍然必须说——为产品经理打造一个产品很难，但我们热爱它的每一点。
