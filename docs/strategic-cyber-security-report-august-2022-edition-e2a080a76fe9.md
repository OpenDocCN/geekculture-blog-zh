*   **Azure 威胁研究矩阵**

> ATRM 主要关注 AzureAD 和 Azure Resource TTPs。由于 AzureAD 被其他产品(如 M365)使用的性质，有时有必要包含也适用于其他产品的技术或技术细节。
> 
> ATRM 的目的不是取代米特 ATT&CK，而是成为纯粹 Azure Resource & AzureAD TTPs 的替代方案。

最后，ATMR 是自由和不可知论者:

> Azure 威胁研究矩阵是与产品无关的，这意味着特定的检测查询默认是针对 Azure 内部的技术，而不是额外的付费解决方案。

*   **数据治理—整体数据保护的 5 个技巧**

微软权限是微软 E5 和 E5 合规性中包含的数据治理平台。客户应利用其数据标签来保护信息，并为所有企业需求提供统一的数据治理平台。

这 5 个技巧是:

1.创建所有数据资产的数据图
2。建立决策和问责框架
3。监控访问和使用政策
4。跟踪结构化和非结构化数据
5。删除不再需要的数据

点击，阅读[的完整文章，了解更多详情。](https://www.microsoft.com/security/blog/2022/08/24/data-governance-5-tips-for-holistic-data-protection/)

*   **新西兰网络弹性框架认证**

什么是新西兰网络弹性框架？

> “这是一个框架或指数，我们正在寻求开发，可用于私人、公共和个人。这不是一个网络安全框架，你有一个勾框，你去，哦，我需要做这些步骤来多一点，”洛佩兹说。

文章进一步解释道:

> CERT NZ 已经开发这个框架大约一年了，在最初的研究中，它还没有在国际范围内发现任何类似的东西。目前只有基于公共信息的索引或系统可用，没有任何东西像政府机构想要开发的那样全面。洛佩兹说，那里存在的许多东西也是相当技术性的。他们不一定考虑网络攻击对个人和社区的社会影响。

CERT NZ 正在寻求帮助，这对希望参与上述框架开发的专业人士和组织来说是一件好事:

> 对于认为该框架有用的企业和行业，Lopez 要求他们联系 CERT NZ。她表示，该机构目前正处于寻求理解问题所在以及如何整合框架或指数的阶段。CERT NZ 希望它对任何人都有用，无论是个人还是企业，以了解他们的行业中是否有特定的弱点或优势。

更多关于 Securitybrief 的[文章。](https://securitybrief.co.nz/story/nz-s-cyber-resilience-framework-to-be-evolving-and-potentially-automated)

# 技术

与技术和威胁相关的一些信息如下:

*   **威胁参与者正在使用新发现的技术来在受损环境中保持持久性。被 MSTIC 发现后，他们将其命名为“MagicWeb”**

> 微软安全研究人员发现了一种我们称之为 MagicWeb 的危害后功能，我们跟踪的威胁参与者 NOBELIUM 使用这种功能来保持对危害环境的持续访问。

MagicWeb 的工作原理:

> MagicWeb 是一种危害后恶意软件，只有在获得对环境的高权限访问并横向移动到 AD FS 服务器后，威胁参与者才能部署它。为了实现他们的目标，即通过验证 AD FS 服务器上任何用户帐户的身份验证来保持对环境的持久访问，NOBELIUM 通过复制 AD FS 操作中使用的合法*Microsoft.IdentityServer.Diagnostics.dll*文件创建了一个后门 DLL。

如何缓解:

> 将您的 AD FS 服务器视为[第 0 层](https://docs.microsoft.com/microsoft-identity-manager/pam/tier-model-for-partitioning-administrative-privileges)资产至关重要，要像保护域控制器或其他关键安全基础设施一样保护它们。AD FS 服务器为配置的依赖方提供身份验证，因此获得 AD FS 服务器管理访问权限的攻击者可以完全控制对配置的依赖方(包括配置为使用 AD FS 服务器的 Azure AD 租户)的身份验证。实践凭证卫生对于保护和防止高特权管理员帐户暴露至关重要。这尤其适用于更容易受到攻击的系统，如带有[登录限制](https://docs.microsoft.com/microsoft-identity-manager/pam/tier-model-for-partitioning-administrative-privileges)等控制的工作站，以及带有 Windows 防火墙等控制的防止横向移动的系统。
> 
> 建议迁移到 Azure Active Directory (Azure AD)身份验证，以降低本地危害横向迁移到您的身份验证服务器的风险。客户可以使用以下关于迁移的参考资料:
> 
> [使用活动报告将 AD FS 应用移动到 Azure AD](https://docs.microsoft.com/azure/active-directory/manage-apps/migrate-adfs-application-activity)
> 
> [将应用程序验证移至 Azure AD](https://docs.microsoft.com/azure/active-directory/manage-apps/migrate-adfs-apps-to-azure)

更多内容请看[官方博客](https://www.microsoft.com/security/blog/2022/08/24/magicweb-nobeliums-post-compromise-trick-to-authenticate-as-anyone/)。
