GitHub Copilot对开发者有威胁吗？(剧透:不是)

原文：https://medium.com/geekculture/towards-abstraction-of-computer-science-cc68e4c30654?source=collection_archive---------47-----------------------#2021-07-28

GitHub最近在技术预告中发布了副驾驶。Copilot是一个基于人工智能的助手，它将从你的应用程序的上下文中自动生成代码，目的是简化软件开发。

随着计算机科学的简化和自动化而出现的一个问题通常是“还需要开发人员吗？”。而每次发生的时候，答案都是【是的】。计算机科学中的抽象降低了进入该领域的成本，从而吸引了更多的人。但是它通常也为新技术、实践和更复杂的系统创造机会。

这篇短文旨在总结计算机科学中抽象层的演变，以帮助理解GitHub Copilot可能产生的影响。

一点历史

最古老和最明智的例子应该是图灵机。它是以破解密码 Enigma 的特定想法创建的，但它背后的概念现在被认为是编程语言算法能力的基础参考。

从那时起，我们创造了处理器，能够随机获取一组指令并进行一般性处理。那些指令最初是用机器代码描述的，这对人类来说很难操作。这就是更高抽象的编程语言出现的原因。

一般来说，我们可以将编程语言的目标描述为编写可以转换成机器代码的人类可读的规范。

编程的演变

编程语言从低级(抽象)语言发展到高级语言，每一次迭代都为通用操作增加了自动化和助手。这些抽象可以采取多种形式，从垃圾收集到库读取CSV文件。

这些发展使开发人员能够增加软件的复杂性，因为技术部分的复杂性降低了。反过来，他们可以为其他人提取更多。得益于此，随着时间的推移，从事编程变得越来越容易。我们甚至看到一些编程语言使用几乎自然的语言(例如 Gherkin )并且几乎不需要任何计算机科学知识来编写。

这并不意味着人们会停止使用抽象层次较低的语言。相反，即使它们通常更难掌握，更冗长，但对于某些用例，例如内存管理，它们可以更加灵活。

我们还可以注意到机器学习的出现，它通过手工或特定代码对分析起来很乏味的数据进行动态处理。在某些情况下，这使得用几行代码编写潜在的复杂行为成为可能。即使这样，也有一些自动化的T2，你只需要提供数据就能得到工作结果。

也就是说，对底层有最基本的了解对于真正理解代码执行仍然非常重要。否则，将很难适应可能和将要出现的问题。

基础设施的演变

我们讨论了代码，但是同样的进化也发生在服务器上。起初，公司从个人电脑开始，直到他们能够在数据中心找到一些空间，租用一台或多台裸机服务器。这非常昂贵，并且需要安装和部署流程方面的技术知识。

虚拟化技术发展到现在，你可以通过云提供商在任何地方的物理主机上租用虚拟机。这极大地降低了成本并增强了可访问性，但是仍然需要技术知识来正确地管理部署管道上的机器。

最近以来，集装箱化技术被大量使用在强大的工具上，如T2码头工人和T4库伯内特斯。这些工具可以帮助您在虚拟机上部署应用，而无需掌握底层系统*。与机器学习一样，还有一些项目旨在进一步减少使用这些技术所需的知识。

最重要的是，一个叫做无服务器的新抽象层正在出现。这使您能够直接在远程容器上执行代码，而根本不必考虑托管问题。虽然这目前主要用于无状态操作，但围绕这个生态系统的用例及工具的增长是毫无疑问的。

*注意:这些工具可能更好地与底层知识结合使用，但是您可以将您的学习重定向到更实际的熟练程度。

副驾驶未来一瞥

GitHub Copilot是软件工程中一项非常有趣的进步，尽管目前还不完全切中要害。根据FAQ，系统在大约50%的时候获得正确的生成，建议对生成的代码保持警惕。

如果我们在GitHub Copilot上看起来有点不同，我们可以将它视为一种新的编程语言。这种语言的输入是文档和函数名，它们被转换成另一种编程语言的源代码。

今天这一代仅限于方法，但我们可以想象这将发展到类甚至包。我们也可以想象这一代人对于众所周知的用例变得近乎完美。

即使有一天一切都在更大的规模上完美地工作，生成的代码仍然需要开发人员处理它所采取的方向，并将更大的图片连接在一起。

结论

关于我们刚刚看到的，毫无疑问GitHub Copilot将进一步抽象开发和简化软件的创建。这一突破有能力推动开发行业，就像以前其他抽象层所做的那样。

话虽如此，开发人员可能永远不会缺少机会，因为由于这个新的抽象层，项目的范围将变得更大。作为开发人员的日常工作可能会改变，但开发人员在很长一段时间内仍然是必需的。

相比之下，我们已经看到系统管理职位转移到了SRE，仍然致力于提供可靠的硬件，但规模前所未有。同样，开发人员可能会进化到更接近的程度，适应新的工具。
