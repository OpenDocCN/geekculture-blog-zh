# 介绍

老实说，许多开发人员讨厌在冲刺阶段结束时进行回顾。

通常，团队成员觉得这是浪费时间:30 分钟的讨论没有任何变化，只是一个在敏捷过程中必须完成的会议。

在这篇短文中，我提供了三个简单的建议，告诉你如何让可怕的回顾活跃起来，并在团队的工作中实现真正的改变。

# 1.有备而来

回顾不仅仅是在冲刺阶段结束时的 30 分钟会议；相反，这是一个连续的过程，记录在两周的时间里什么是对的，什么是错的。

通常，团队成员很难在回顾中提出任何有用的东西:他们记不起关键的细节，或者缺乏论证某个观点的例子。

如果这是一个反复出现的问题，我建议每个团队成员在他们电脑上的笔记应用程序中保留一个笔记部分。

一旦两周冲刺中出现任何值得在即将到来的回顾中提出的事情，都应该立即记下来，以免被遗忘。

在回顾中，这就变成了一个无压力的练习，打开他们的笔记应用程序，把这些点转移到便利贴上。

# 2.创建行动任务

除非商定后续行动，否则还会犯同样的错误。

解决方案是在**的下一个冲刺**中创建一个*杂项*部分，并给人们分配名字。

例如，假设在持续集成阶段，由于 SonarCloud(或类似的应用程序)中的代码质量检查失败，拉请求被提出，然后被阻止。

这个问题的解决方案是，所有开发人员都需要在他们的 IDE 上安装 SonarCloud 扩展，以便在开发人员提出拉请求之前标记出质量问题和漏洞。

因此，需要在 Misc 部分**中为每个开发人员**创建一个任务，并将他们的**名称分配给任务**以安装 SonarCloud 扩展。

# 3.增加一些刺激

你需要想出一些点子，给回顾会注入一些乐趣，让它们不那么无聊。

我尝试的一个成功的想法是将每个人的名字放在一个[旋转轮应用程序](https://tools-unite.com/tools/random-picker-wheel)上，这样当旋转轮停在某人的名字上时，就轮到他们通过他们创建的便利贴与团队的其他人交谈。

是的，我知道这听起来有点傻，但是试一试吧！
