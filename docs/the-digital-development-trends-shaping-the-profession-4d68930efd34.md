# 塑造职业的数字化发展趋势

> 原文：<https://medium.com/geekculture/the-digital-development-trends-shaping-the-profession-4d68930efd34?source=collection_archive---------13----------------------->

![](img/bc7c0e88745b58c43e0fe516b761fc2d.png)

当浏览器博客团队让我为 2022 年之交写一篇“数字发展趋势”的帖子时，我起初并不确定。我试图解释说，在很长一段时间内出现的大多数发展宏观趋势不会在一月份轻易下降——用一句老话来说，是进化而不是革命——任何文章都需要更深入、更深远。

“听起来很棒！”这是他们的回复(不过，他们最终还是找到了一位同事来写那篇聚焦 2022 年的文章，[，这里是](https://www.browserlondon.com/blog/2022/01/10/5-front-end-development-trends-for-2022/))。所以在这里，我们谈谈我对发展职业的长期发展方向的一些想法。特别是，我看到了易于配置的 SaaS 产品对项目角色的影响，以及开发人员对变化保持开放态度的必要性。

TL；DR:编码不再仅仅是开发者的事情了。我们将会编写更少的代码；而且，我们编写的代码将是特定于领域的。

# 发展角色开始变得模糊

贯穿软件世界的传统思维通常是谁负责什么的结构:开发人员对非开发人员，后端人员对前端人员。但我有一种感觉，我们正从这种心态转向一种更加灵活、以结果为导向的方法。越来越常见的是，设计师在他们的投资组合中有开发工作，并希望实现自己的想法，以及新的 MBA 毕业生进入组织编写自己的 python 脚本来获得他们想要的数据。还记得我们常说的“每个人都应该学习编码”吗？他们做到了。

低代码和无代码解决方案也影响着宏观数字发展趋势。像 Wix 和 Squarespace 这样的 web 平台上的配置选项真的很强大。虽然今天的营销经理可能正在学习编码，但今天的开发人员可能会选择配置现有的服务，而不是编码来获得最实用、可持续的结果。我和一位同事最近在收到提案后的 5 周内向公众推出了一个客户项目。我写的唯一代码是大约 15 行 CSS，以获得正确的品牌，其余的只是将一些服务连接在一起。

对于较小的项目，客户更希望每个开发人员都有足够的能力来处理前端和后端。设计和前端开发角色之间也有一些模糊，特别是通过配置可以实现的结果。我可以看到一些障碍正在消失，专业开发人员的定义变得越来越模糊。

我会鼓励开发人员不要过分看重他们的角色或职责。在这里，关键的元素将是确保无论你的项目涉及到什么角色，你的利益相关者都是中心。项目的成功是最重要的结果。你的自我应该是一个非因素。尽量不要担心谁的工作角色中有“开发人员”这个词。

# 有一种 SaaS 产品可以解决这个问题

康威定律指出，当一个组织试图设计一个系统时，它最终会反映出该组织的沟通结构。尽管这条“定律”已经存在了 50 多年，但我认为它与我在 CMS 和 web 开发领域看到的越来越多的事情非常相关。那些庞大、单一和复杂的 CMS 或数字体验平台开始无法满足营销和 IT 团队所需的灵活性、敏捷性和便利性。

如果你觉得你在和你的 CMS 战斗，那么它的日子一定屈指可数了。一种更精确、更有用的产品[，通常以 SaaS 品牌](https://www.browserlondon.com/blog/2019/12/02/digital-product-development-build-or-buy-services/)出售，将更好地填补空白，减少任何摩擦；对于开发社区来说，这意味着更少的编码和更多的配置。康威定律可能是对我们如何设计软件以满足不设计软件的人的期望的控诉，但不同的解读可能是有启发性的:“嘿，填补会计和项目经理之间的沟通差距”，而不是“嘿，重新设计我们业务的沟通结构以适应 WordPress 允许的”。

![](img/fb0231ae7ddc87dd121ea46538a992af.png)

SaaS 的产品已经开始扼杀瀑布方法，这种趋势只是加快速度。其中一些产品的质量很高，这意味着不再有理由默认瀑布式项目，因为您有后端开发人员、前端开发人员、两个月的时间表和可观的预算。相反，你可以在两周内准备好一个强大的 [MVP](https://www.browserlondon.com/blog/2021/09/17/dont-run-before-you-can-walk-benefits-mvp-build/) ，即使它不能马上做好你需要的一切，你也可以随后增加它。也许[敏捷方法论](https://www.browserlondon.com/blog/2015/05/28/the-agile-process-explained-for-non-technical-people/)的承诺是在解决方案管理而不是软件工程中？

我认为，作为开发者，我们需要保持灵活性，专注于什么对网站、内容、体验和预算最有利。这有时意味着不使用 CMS 或我们已经习惯的方法。这听起来可能很明显，但它是关于正确工作的正确工具，以及正确项目的正确产品。今后，我们将与更多不同的 SaaS 产品合作。

# 编程语言正在转变

好了，到目前为止，我主要是从编码以外的领域谈论数字发展趋势。但是，嘿，编码呢？嗯，它似乎不会消失。相反，它正被整合到其他学科中。但也许“网络开发者”可能会离开。我知道我已经不再用“web 开发人员”来形容自己了，除非是在晚宴上，我需要一个“我编写供互联网上的人使用的代码”的简写 2014 年， *web 开发人员*是最热门的工作。到 2024 年，LinkedIn 的摘要*“网络开发人员”*将简单地表示“*”，而尚未指明专注于代码的实习生将在某个行业开始工作*。但是我们是怎么到这里的呢？

## 让我沉浸在一点怀旧中:还记得 web 2.0 吗？

宏观趋势有一种半周期性。或者可能是一种潮起潮落？一些变化(如智能手机的引入)导致了市场范式的转变，新领域的所有竞争对手都出现了，然后发生了某种演变，市场稳定下来，分为赢家和输家，然后范式再次转变，整个事情重复进行。

考虑到这一点，如果我们想了解这十年的前五年，那么思考过去十年的前五年是很重要的。在本世纪初，我们刚刚经历了网络技术的寒武纪大爆发。像 Ruby on Rails 这样的新框架正在生根发芽，而且有关于服务器端 JavaScript 的传言；像 Android 和 iPhone OS 这样的新目标，加上谷歌开发的新浏览器；以及移动优先(或仅移动)等新业务需求？！)应用程序。每个新的 web 应用程序的范围——几乎所有的 web 应用程序都是新的——都出现了爆炸，Bootstrap 和 Foundation 这样的前端框架将使每个人都有可能同时服务于所有的应用程序——“一次编写，随处运行”变得显而易见，就像 Java 在 90 年代承诺的那样。

发生了什么事？本周最热门的新 JS 框架去哪里了？嗯，这很难说，但我会试试看。

## 我们生活在一个稳定的岛上

令人欣慰的是，虽然痛苦，但定义了过去十年的非理性繁荣偶然发现了软件开发的世俗现实，市场不可避免地收缩和稳定，有利于更具生产力的产品。在 JavaScript 世界中，VueJS 和 [React](https://www.browserlondon.com/blog/2021/02/09/react-is-king/) 与它们尚未灭亡的竞争对手(Ember、Knockout、Batman、Meteor、AngularJS、Backbone、lodash)一起诞生，早在 2014 年，Ruby on Rails 就被宣布死亡，预测它即将死亡仍然是每年的传统。本质上，如果你的框架不能满足足够多企业的需求，那么它就会被满足需求的企业淘汰。导致了我们今天的处境。

![](img/22298b0a2eae8fe42964e7e2fb07a931.png)

在当今的开发世界中，一切都很好。我所说的“好”是指稳定、高性能、成熟和相对简单。

尽管在 JavaScript 世界中，life is good 很早就名声在外(无论是否赢得)。无论喜欢与否，Typescript 都是好的。不管你喜不喜欢，NPM 是好的。VueJS 和 React 都不错——选一个，开心点。我们现在已经看不到这么多的 JavaScript 框架了，因为今天的 JavaScript 世界是“挑你的喜好”而不是“挑你的毒药”。

在 PHP-land 中也可以这么说，那里的生活也很美好。拉勒维尔和 T2·西弗尼都很开心。对不起，失败者，但是开发人员将会去有工作的地方，工作板上的 Drupal 和 CodeIgnitor 部分正在减少。(这里奇怪的异常值总是 WordPress 的 Schrodengerian 状态，即 good 和 not good)。)PHP 本身不再缺乏所谓的关键特性，最近的大多数更新都与生活质量的提高有关。

选择一种语言，你会发现我们生活在一个充满可能性的时代，这要归功于定义[赢家和输家](https://www.browserlondon.com/blog/2021/02/09/react-is-king/)的无情进化过程。如果你想建一个网站， [web app](https://www.browserlondon.com/blog/2020/06/08/web-site-vs-web-app-whats-the-difference-does-it-matter/) ，混合 app，或者别的什么，你需要的工具都有。当我们考虑工具时，除了稳定性和安全性，没有更多的选择余地。我们知道什么是可行的，并且有大量合格的、有经验的开发人员来开发它们。事实上，您可能不需要像前面讨论的那样为许多工作提供高质量、现成的解决方案。

## 我们能谈谈明天吗？

有些东西是脚本语言无法提供给我们的。你在和其他开发者的对话中注意到了吗？

你上一次看到非传统脚本语言(即，不是 PHP、Python、Perl、Ruby、JavaScript 等的守旧派)大规模进入市场或引起年轻开发人员的兴趣是什么时候？几年前，在你最喜欢的技术会议上，主导主题演讲的话题是范式:[函数式编程](https://www.browserlondon.com/blog/2019/06/10/functional-css-perils/)与面向对象编程(OOP)。我们似乎对那次谈话不再感兴趣了。所有这些讨论的结果很简单，从一开始就应该很明显:遵循你的语言最适合的范式，并根据什么最适合你的产品来做出决策。因此，如果对话不是关于范式，那么社区想要谈论什么？

过去十年的数字发展趋势集中在框架上，而这十年正在形成以语言为中心的趋势。当 web 开发人员从一个框架跳到另一个框架时，同时也有一个不那么下意识的平台——本地语言革命正在酝酿中。具有严肃社区和严肃支持者的新的专门构建和以任务为中心的语言开始出现。

压倒性地，谈话已经转向了[强类型语言](https://stackoverflow.com/questions/2690544/what-is-the-difference-between-a-strongly-typed-language-and-a-statically-typed)。我认为我们正朝着四、五种主要的强类型、多用途语言的方向发展，它们正在走向成熟。TypeScript、Rust、Go、Kotlin 和 Swift——其中一些有主要支持者，包括 Google、Mozilla、Apple 和我的 Patreon 帐户——可能会成为未来几年的流行语言，以及一些优秀的相关 web 框架，如 Rust 的 Actix-web 和 Swift 的 Vapor(像 Kitura 一样不可避免的失败者，但仍然是一个精彩的项目)。关注[年度栈开发者概览](https://insights.stackoverflow.com/survey/2021)会很有意思，看看编程语言的流行程度在未来两到三年内会如何演变。

我将要关注的语言有一些共同点，尽管它们来自不同的地方:Go 是为服务器构建的；Swift 是为 iOS 开发打造的；Android 版 Kotlin 为浏览器生锈；用于 web 的类型脚本。都是强类型的。所有这些都是专门建造的。所有这些都是稳定和强大的，然后才迈出他们的小步，预示着编程新时代的到来。我们将看到每一个人都尝试成为这十年的 Vue 或做出反应。

为了理解未来会是什么样子，我们需要开始想象任何给定语言的胜利条件是什么。例如，如果你相信 Web3 (1)是真实存在的，并且(2)在公众最终对被 NFT 和其他东西——硬币——诈骗失去兴趣之后，它还会存在，Rust 就有迄今为止最好的故事。类型安全和运行时保证(如无竞争条件)提供了对智能契约设计至关重要的承诺。Rust 在这一领域最合适的竞争对手是 TypeScript，它为 JavaScript 开发人员提供了一定程度的安全性，让他们可以快速搭建原型。谁会赢:训练有素的 Rust 开发人员，还是敏捷的 TypeScript kiddy？无论哪种方式，这将是一个强类型系统，使他们-如果只是为了品牌。谁会输？我猜是实时编译器。

到目前为止，我提到的每种语言都有一个故事，为统治世界提供了一个令人信服的故事，类似于我刚才对 Rust 的描述。

## 什么会出错？

这些不断发展的语言和一些相关框架的一个重要元素是确保它们有正确的管理，以产生对其长期未来和方向的信任。在这里，那些大支持者通过开源做了正确的事情。例如，当苹果开源 Swift 时，他们向社区提供的不仅仅是当前版本，他们一直追溯到最初提交的版本。当这些语言是专门构建的，但又是社区驱动的时，它们的长期未来是有保证的——围墙花园式的方法既不会产生长久性，也不会获得开发者的信任和好感。

苹果可以对 Swift 进行一些控制，但 IBM 和公众(以及其他人)贡献很大，有相当大的影响力。一个用各种语言重复的有成功可能的故事。Mozilla 已经放弃了对 Rust 的控制。

你有时会看到框架处于看似永久的测试阶段，版本之间的步调太长，或者方向受到社区的质疑。这些框架正处于衰退的危险中，并陷入“业余爱好领域”。信任是一个框架成功的基础，而社区信任会以惊人的速度衰减。Rust 的 actix-web 在这一领域取得了成功(尽管围绕版本 2 有一些争议)，Rust 的 Rocket framework 却彻底失败了，因为它从未完全准备好下一个测试版的次要版本(目前 0.5-rc1 已近一年)。

# 概括起来

软件开发和后端开发的角色总是在不断发展。未来几年的宏观数字发展趋势将意味着我们使用的语言、我们用于项目的产品以及我们扮演的角色都将发生变化。成功的个人和团队将是那些能够适当灵活地为客户提供最佳价值和结果的人。

把你的精力集中在最有成效的未来上，你会挑选出赢家，并试图避免过去十年中因过快涉足新技术而犯下的愚蠢行为。早期采用者支付最大的罚款，而晚进入通常意味着有大量的文档和堆栈溢出线程需要处理。
