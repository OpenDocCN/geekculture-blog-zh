随着这些年来手机的速度越来越快，我们很久以前就已经达到了可以运行普通版本的网站而不会出现任何问题的地步。然而，让你的客户在他们的手机上安装你的应用程序有一个明显的优势。首先，它们总是离你很近。

开发移动应用程序的问题有很多解决方案，因此我们决定一个一个地为您介绍。我们要解决的问题是，您的企业想要开发一个跨平台的移动应用程序。你该怎么办？

# 渐进式网络应用(PWAs)

我们知道，我们知道。它们在技术上不是移动应用。然而，我们将把它们算作本地 API，因为越来越多的本地 API 可用于 web 应用程序，它们是可安装的，并且可以离线运行。不只是我们也这么想。很少有人知道 PWAs 是史蒂夫·乔布斯在 2007 年的演示中为 iPhone 应用程序设想的。14 年前，苹果 CEO 预见了未来。

当然，我们知道事情的结果——苹果的确发布了一个 iOS SDK 来开发“原生”应用程序，2008 年推出了应用商店。福布斯认为最初加倍投资网络应用技术的决定[是一个“大错误”。当然，《史蒂夫·乔布斯传》可能会被认为是有争议的。当时，手机还不够强大，无法实现开发人员使用 JavaScript](https://www.forbes.com/sites/markrogowsky/2014/07/11/app-store-at-6-how-steve-jobs-biggest-blunder-became-one-of-apples-greatest-strengths/#4b471804652b)[的设想，而当时这种语言也没有处于最佳状态](https://www.itmagination.com/blog/the-complete-history-of-javascript-typescript-and-node-js)。

然而，14 年是一段很长的时间，我们确实看到了这一点——手机经常“摇摆”八个核心，6gb 的 RAM，能够运行要求苛刻的软件。今天，他们已经准备好了。这位苹果公司的传奇 CEO 简直是走在了时代的前面——可惜他没能活着看到自己的愿景变成现实。一丝希望来自火狐操作系统——一个小众且停产的操作系统，只有用 JavaScript 编写的应用程序。

# 那么，公共福利援助有什么好处呢？

## 代码可重用性最大化

通常有一个关键问题——是先开发 web 应用程序还是先开发移动应用程序。PWAs 提供的答案是…为什么不两者都要呢？

创建共享 100%代码的应用经常是现实。没有理由调整你的代码。[你可以提示用户在 Chrome 运行的所有平台](https://web.dev/promote-install/)上以同样的方式安装你的应用**——Linux、Windows、macOS、Android、iOS……用一个代码库。**

## 没有应用程序下载

这个标题可能会让人产生误解。如果很明显，一个人必须为网站下载代码，怎么可能没有应用程序下载呢？这是真的。你的用户仍然需要下载你的网站。然而，在这种情况下有一些关键的不同。

*   您正在减少安装应用程序所需的步骤
*   应用程序的大小要小得多
*   这款应用更容易被发现，你的用户在 Play Store 或 App Store 中看不到竞争对手的广告

## 更快上市

在你的产品上市之前，很少有比预期上市时间更重要的指标。一个真正的祝福是能够减少它，同时不会在你的解决方案的质量上妥协太多。尤其是，所有团队成员都可以专注于一个目标——完成 web 应用程序。

## 平台独立性

苹果在很大程度上(谷歌在较小程度上)通过 App Store 和谷歌 Play 商店垄断了 iOS 设备上的软件分发市场。为了让手机对所有人都具有无懈可击的安全性，对于 iPhones 上的侧载应用程序没有真正的解决方案。在官方商店之外的 Androids 上安装应用程序要容易得多，尽管大多数用户仍然选择官方方式获得应用程序。

大大小小的公司( [Epic、Spotify、Basecamp、Protonmail、Deezer、OpenDataBot 和其他公司](https://appfairness.org/))意识到了这一点，并在一年多前组成了一个“[应用程序公平联盟](https://appfairness.org/)”。它的目标是“倡导整个应用生态系统的选择自由和公平竞争。”

有一种方法可以分发你的应用程序，由你控制，你可以从中获得所有的利润，并且使你的应用程序很容易被发现。这是什么？**你的网站。**pwa 不像原生应用那样面临那么多障碍，这是一大胜利。

## 轻松更新

由于必须在网络浏览器中运行，你的应用程序可以在用户每次上网时更新。这提供了一个轻松快速的更新过程。您可以通过缓存(缓存=将应用程序保存在临时位置以便快速访问)配置来控制更新策略。您可以选择为您的应用提供服务:

*   仅缓存(不是最佳方式)
*   仅网络(不是最佳方式)
*   缓存退回到网络(离线优先应用)
*   网络回退到缓存(频繁更新的应用；在网络连接不稳定的情况下，该过程可能需要较长时间)
*   先缓存再网络(更新频繁的应用；更好的用户体验)

你可以在这里看到所述方法[的细节和示例实现。](https://developers.google.com/web/ilt/pwa/caching-files-with-service-worker)

**当然，这种解决方案肯定有一些缺点，对吗？是的，有。**

# 使用 pwa 的缺点

## 比原生应用程序慢

史蒂夫·乔布斯和苹果公司放弃全面拥抱网络技术的主要原因是缺乏动力。速度不够。开发人员不得不改用 Objective-C。

对于安卓系统，选择的语言是 [Java](https://www.itmagination.com/blog/java-17-new-changes) 。这种方法允许创建更快的应用程序，也许更重要的是，为谷歌和苹果带来了稳定的资金来源。

尽管现在的手机与一些台式电脑的性能不相上下，但是本地应用或者混合应用才是更受欢迎的解决方案。一个原因是执行的速度。无论如何，一个运行在浏览器上的网站即使在最快的电脑上也会变慢。

随着接口以 60fps 流畅运行，人们不得不怀疑这是否仍然是放弃 PWAs 的有效理由。

## 一些本地 API 仍然不是内置的

尽管 Chrome 可以:访问通讯录，访问原生文件系统，实现 NFC 解决方案(NFC 是如何工作的，例如非接触式卡)，以及处理蓝牙连接；

仍然有许多事情它不能做。多亏了项目 Fugu，web 开发人员不能做的事情越来越少，幸运的是，越来越少。

然而，有时候，开发人员不得不使用本地 SDK 来解决他们的问题。

## 人们不习惯在浏览器中安装应用程序

一条“用户体验法则”说:

> *用户大部分时间花在其他网站上。这意味着用户更喜欢你的网站像他们已经知道的其他网站一样工作。*

实际上，这也适用于你希望用户安装应用的方式。如果你的大多数竞争对手，甚至是在完全不同的领域运营的公司，他们的应用程序与你的不同，你的用户可能会有点困惑。

这是公司在完全接受 web 优先方法之前必须面对的最大障碍。这个问题也没有明显的解决方案。

# 我应该何时决定 PWAs？

谷歌试图让所有重要的原生 API 都可以通过 Chrome 使用，初创公司肯定是最大的受益者。如果你有一个相对较小的软件开发团队，你将是最幸福的。同样的道理也适用于你的应用会快速更新的时候；没有比创建 PWA 更快更新应用程序的方法了。
