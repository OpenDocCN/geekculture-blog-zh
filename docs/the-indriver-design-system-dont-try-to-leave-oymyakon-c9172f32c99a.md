## 在本文中，来自 inDriver 的 iOS 开发人员 Alexey Kakoulin 分享了他在设计系统处理方面的经验。

![](img/1004b277612c03b6417000a1785671b8.png)

我们的应用程序是一个超级应用程序，提供各种服务和资源:同城、城际和货运旅行，以及快递、勤杂工服务等。每个服务都部署了独立的专门团队，他们可以自由地做出决策或考虑几乎任何产品开发选项。

但是让我们想象一下，如果给他们完全的设计自由，会发生什么。我们会得到什么结果呢？混乱(极有可能)。这就是为什么我们有一个系统设计团队，我们想告诉你更多关于它的信息。下面，我们的 iOS 开发人员阿列克谢·卡库林(Alexey Kakoulin)详细谈谈这一点。

# 背景

让我们假设我们有一个抽象的任务，例如，设计一个“创建订单”按钮。第一小组负责这项工作。他们完成了工作，现在添加了新按钮。过一会儿，团队 2 也需要一个按钮。他们着眼于团队 1，但决定以自己的方式设计他们的团队。但也有第 3 和第 4 团队，他们看到这种情况，但认为，“这一切都很酷，很好，但我们会用自己的方式来做。”

但是由于它们都在同一个应用程序中运行，我们将会得到一个不一致的设计元素的混乱组合:

![](img/bf02335fa1e3fd5b6286eb316e82a6ea.png)

Each team reinvents its own wheel

为了防止这种情况发生，inDriver 在 2021 年提出了一个名为 *Oymyakon* 的设计系统。它是以“寒冷的极点”之一命名的该地区以严重的霜冻和低温记录而闻名，但它也拥有美丽独特的风景。

![](img/e7b96ea81118dd7e116f88d4fefc2f06.png)

Officially recorded temperatures in Oymyakon ranged down to minus 67.7 degrees centigrade (Celsius)

名称选择过程漫长而紧张，涉及多次反复。最初的努力包括在开发社区中进行调查，但是没有任何有用的结果。然后我们想出了*设计系统*这个名字，听起来太陈腐、太老套了。

最后，当一个成熟的团队成立时，我们的人进行了一次全公司范围的投票，选出了最好的建议，并进行了投票，最终*奥伊米亚康*胜出。

设计系统团队目前由四名移动开发人员、两名设计人员(针对 web 和移动设备)、一名测试人员和一名负责处理最具挑战性问题的领导组成。所有这些都是我们移动集群的一部分。

还有一个 web 集群，它有一个平台团队，在平台任务之上，还负责设计任务。这些人为我们的产品团队提供“构建模块”。

# 什么是设计系统？

每个人都以自己不同的方式描述一个设计系统。其中一份报告中有这样一段冷静而贴切的描述，将其比作一个多层蛋糕:

![](img/179d114ab0f16410fe9a607c9d3e3807.png)

This is the very same cake

因为人是这里的基本要素。接下来是围绕人构建的流程，以及这些人创建和使用的组件。所有这些都覆盖了厚厚的一层文件，使它更加甜蜜。

设计系统团队的目标是快速找到组件，及时更换，提高品牌知名度，给用户带来喜悦和快乐。用户不仅是下载应用程序的最终用户，也是使用组件的开发人员和设计人员。

为了实现目标，一切都分为两个部分，即工具和过程。

# 工具

设计师在 Figma 绘画和工作是真实的来源之一。Figma 是一个成熟的文档，拥有自己的模板、设计规则和任务设置示例。

一切自动卸载，并与 Figma 同步。不同的平台使用不同的工具。例如，Figma Export 用于 iOS 和 Android，这是一个由 Swift 中的 iOS 专家编写的开源库。这允许您快速进行编辑并参与开源社区。

![](img/835593f138238a229e0f346b9e6c3bb7.png)

下一个要使用的解决问题的工具是组件库。对于移动应用，它是一个演示应用，而对于网络，它是故事书。该工具允许任何人快速查看组件:设计师、开发人员、产品经理或技术领导。该工具可用于查看组件、代码样本或检查布局，以及进行像素完美测试等。

![](img/304f9968275724c8932c2dbf0ab508e7.png)

Gallery specimen

为了确保没有人会意外损坏组件，快照测试涵盖了所有组件。

当然，团队写文档，有很多，这是一个重要的过程。很少有人喜欢这样做，但是当团队发展非常快时，这种方法使得进一步发展公司和补充流程变得容易得多。

# 处理

文档很重要，也可以归类为过程。但是沟通更重要。

这一切都从入职开始，当一个新的团队成员第一次加入团队。他们被告知设计系统由什么组成，它在哪里，在哪里找到你要找的东西，如何阅读一个主题，以及向谁寻求帮助或有问题要解决。

设计系统团队积极参加内部会议。对于每个平台来说，它们是不同的，每周举行一次或两次，取决于具体相关人员表现出的主动性。在那里，我们谈论我们所学的，或者我们回顾以前的学习，以防止它从记忆中消失。

设计师每周对创建的布局进行审查。在那里，设计社区审查加入设计系统的新候选人，以及审查和提出尴尬的问题。例如，如果一个组件看起来像设计系统中的一个组件，但由于某种原因没有从那里取出。或者争论他们的小发明，按钮的圆角等等。

该团队以两周冲刺的方式工作。一旦完成，公开的 sprint 评审将被举行以评估结果，任何人都可以加入并提出问题。在那里，所有参与者都报告他们完成了什么或没有完成什么，为什么，以及他们在前进的道路上遇到了什么困难。如果有人不能来，会在内部聊天和公司门户网站*队友*上发布一段视频。

由于设计系统以开放的过程为前提，团队之间的互动是必不可少的。每六个月，团队收集一次反馈，评估其表现和结果，并得出结论，判断是否朝着正确的方向前进。

# 是否需要单独的设计系统团队？

要回答这个问题，问自己三个问题:

1.你的成长速度有多快？如果你有多个团队，他们有自己的设计师、产品经理、技术负责人、愿景，并且你确实希望他们在不被日常任务分散注意力的情况下向用户交付功能，那么就值得对这个问题进行充分的思考。

2.你多久进行一次设计变更？设计变更经常发生。如果您在更换基本组件时遇到问题，请仔细考虑这个问题和可能的解决方案。

在过去的 10 年里，inDriver 已经经历了三次重新设计的过程。如果说前两次进行得或多或少还算顺利，因为只有一个团队，整个过程没有任何问题，那么第三次就很艰难了。随着团队数量的增加，需要付出巨大的努力来实现预期的结果。

![](img/ac8d4554becff1f4f30f62f990798222.png)

Three redesign campaigns at inDriver

3.开发商和设计师之间会产生纠纷吗？当然，随着设计系统和团队的到位，它们会变得不那么频繁，但不会完全消失。要考虑的重要一点是:如果一开始开发人员为设计人员工作，那么设计人员将为开发人员工作。

这是非常重要的:如果设计者犯了一个错误，忽略了一个与设计系统相背离的布局错误，开发者总是可以停下来指出这个缺陷。然后设计师会返工，这真的有用。

# 快速回顾一下

在本文的最后，我们将分享 inDriver 的设计系统在不久的将来会发生什么。

1.共享组件数量的增加。到今年年底，他们将有 50 人。除此之外，还有计划使用苹果和谷歌提供的新框架，即 Swift UI 和 Compose。

2.提高服务可用性。目标是让残障人士也能使用该应用程序。部分屏幕已经进行了调整，我们正在努力确保该应用程序在年底前完全适应。

3.团队实验的自由。设计系统很酷，但是团队想要自由。设计系统团队想要给他们，但是监控检查需要作为某个过程的一部分来执行。

4.另一场重新设计的运动即将到来，这是第四次了。我们要补充的是，大约六个月后，inDriver 应用程序将被更新得面目全非。
