# 介绍

我在数据领域工作了大约十年。在那段时间里，我参加了很多面试，也主持了一些面试。我已经看到了一些可以轻松避免或改进的模式。有人说数据是新的黄金/石油，但我非常喜欢培根，所以我们姑且把它视为数据是新的培根。公司要向前发展并利用数据，就需要优秀的数据工程师。找到一个伟大的数据工程师几乎就像一次探险，因为一个伟大的数据工程师几乎就是一只独角兽。我决定就如何成为一名伟大的数据工程师，或者为了达到这个目标，或者至少为了脱颖而出，你应该关注哪些方面，发表自己的意见。

# 软技能和你的简历/投资组合

你可能会认为，如果你是一名工程师，你就不需要努力提高你的软技能。在我职业生涯的初期，我注意到有时候你的软技能比技术技能更有价值。在技术上，你可以提高得更快，但在软件上，有时需要心态的转变，这通常不是那么简单。

## 简历中的流行语

**问题**:有时候，在简历中加入一些流行语是很酷的。我很喜欢我在某处看到的一个帖子(或者可能是某处的一次谈话)，在这个帖子中，这个家伙在他的简历中列出了一堆口袋妖怪，并要求招聘人员指出它们。

**解决:**不要胡说八道的人。很容易发现这种情况，你只是触及了一些技术可能性的表面，然后你把自己提升为专家。是的，面试是你对公司的推销，但是如果你被发现说谎或误导，你就会失去信任。如果你想在简历中加入一项技能，它不应该仅仅是基本技能。做一个概念验证，在一些项目中更好地学习它。需要内容？LinkedIn、Youtube 和多个其他网站上的大量免费内容。如果你想更深入，通常一些书会帮助你。

## 不要害怕问问题。

**问题:**在面试中，我最喜欢观察人们是如何思考的。他们问什么问题来理解任务。这看起来微不足道，但大多数人试图强行通过。要么让问题变得过于复杂(就像试图用猎枪或手榴弹杀死一只苍蝇)，要么他们没有正确理解问题是什么，没有要求澄清事实(例如，你在谈论鞋子，他们给你一个关于企鹅的答案)。

**解决方法:**不明白也没关系(我的意思是，如果是超级基本面的问题，那么可能就有问题)，但绝大多数情况下，你可以多问几个问题，直到搞清楚了，或者你认为自己明白了——用文字总结一下，问问自己理解对了没有。如果是类似于现场编码采访(尤其是缩放时间)的事情，并且你正在分享你的屏幕——做你通常会定期做的事情。我喜欢一个家伙，他说——我不记得了，但是我能做到。他打开谷歌，找到相关答案，并将其纳入自己的代码。这是诚实的，它表明你知道如何做这项工作，即使你不能马上知道答案。

## 展示你的项目。

**问题:**有时候，当你进行初步的简历筛选时，很难理解人们的技能水平。就像我之前提到的，最大的问题是流行语，即使你刚刚掌握了它的最初想法，也把它当作一种技能。

另一种情况是，当我在 LinkedIn 上寻找潜在的候选人时，我以特定的方式输入我的搜索，并打开许多个人资料。如果它以任何方式脱颖而出，我会让它稍后开放，在那里我会更彻底地检查它(打开 GitHub，博客帖子等)。).

**解决方案:**在你的个人资料上突出显示你的帖子，添加你的 GitHub 库，这样人们就可以查看你了。有时制作一个网站来代替你的简历会给人留下深刻的印象。请记住，你必须用行动或知识来支持你的话。人们在一英里外就能闻到谎言！总而言之——努力打造你的在线形象/个人品牌。即使你没有寻找新的工作，机会迟早会找上门来。

# 技术

## 结构化查询语言

**问题:**所以，这一条相当直白。无论您是数据分析师、数据科学家还是数据工程师，SQL 技能都是最需要的。大多数人(比如 80%以上)在面试中没有通过 Codility SQL 任务。我是说，没有一个测试案例是超级复杂的。只是检查一下基础知识，也许还有一两个更高级的话题。

**解决方案:**你必须知道你在日常工作中会用到的最强制性的东西——窗口函数、分组依据、CTE。了解 SQL 的来龙去脉——你将离你梦想的工作更近一步。使用带有 SQL 任务的在线平台在那里进行改进，即 [HackerRank](https://www.hackerrank.com/) 、 [W3Schools](https://www.w3schools.com/sql/default.asp) 。

## 数据建模

**问题:**要构建一个健壮的 DWH，你必须知道不同的数据模型，以便依靠最适合这种情况的模型。当然，星型模式总是可行的，但是我怀疑它是否适合所有人。根据我的经验，这取决于具体情况，也取决于你使用的技术。

**解决方案:**阅读 Kimball 建模技术、Data Vault、Inmon 方法以及其他许多相关内容。不要处于你知道的黑暗时代，甚至不要尝试或费心去阅读其他的。如果你愿意，你可以读一读我前段时间写的关于数据建模的[高级介绍](https://towardsdatascience.com/guide-to-data-warehousing-6fdcf30b6fbe)。如果你决定深入某个特定的领域，它会有更多的来源。

## 编程语言

**问题:**数据工程通常被认为是一个大量使用 python 的领域，人们有时会因此而看不起数据工程师。大部分 DE 都是用 python 完成的，比如 Airflow 和 Luigi 这样的 orchestrators 都是用 python 构建的。你可能听说过熊猫和其他漂亮的图书馆，它们在 DE 社区里很受欢迎。但是就拿 Spark 来说吧，它在大数据生态系统中超级重要。它构建在 JVM 上，但是有一个 python 接口，以便更好更快地采用。在 Scala/Java 中使用它有自己的用例，有时性能会更好，因为它是一种 JVM 语言。

**解决方案:**了解任何语言的基础和面向对象的方法，其他一切都会水到渠成。请不要拘泥于一种语言；有时候，让所有的栈只绑定到 python 是没有意义的。也许你想利用强类型语言的优势，在编译时出错，而不仅仅是在运行时——与团队讨论并尝试引入它，看看它是否可行。有时候你需要不同的工具来做好工作——你不会用螺丝刀往墙上钉钉子，对吧？

## docker/虚拟环境

**问题:** Docker，Kubernetes 才是未来。旋转本地隔离环境将帮助您更快、更有效地开发，而无需在 prod 上运行。非常容易在本地启动任何数据库的一个小实例，插入一些测试数据，并在较小的规模上更快地开发管道。python 虚拟环境也是如此。管理依赖项，让它们安装在本地，这是一个棘手的问题。使用虚拟环境(甚至是 docker 映像，至少 VSCode 和 PyCharm 也能很好地使用它们！)来收拾残局，有更多的结构。

**解决方法:**学习它的基础，使用它，掌握它。我推荐 Docker 上一个精彩的[实践教程和 Kubernetes](https://www.freecodecamp.org/news/the-docker-handbook/) 上一个类似的[实践教程。它将提供关于这些工具的极好的视角和一些使用它们的经验。但是我强烈建议你看完之后，不要把它们当成你的技能。这只是最基本的。](https://www.freecodecamp.org/news/the-kubernetes-handbook/)

## 一般数据区

**问题:**我认为每天都有处理数据的工具和框架诞生。你必须快速适应不同的工具和总体环境中发生的事情。我看到许多人说他们使用 Apache Spark 1.6、Sqoop 等。有些已经发布了更新更好的版本，或者它们已经被其他更高性能和更新的工具所取代。

**解决方案:**在 Twitter、Linkedin 上关注那些分享大量新闻的人，对能够完成工作的环境和工具有一个基本的了解。努力跟上市场，尝试新事物。
