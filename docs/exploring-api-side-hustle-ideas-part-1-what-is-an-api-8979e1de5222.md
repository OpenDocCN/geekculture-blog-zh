# 一些背景

像我们大多数人一样，去年我的空闲时间比计划的多得多。我不顾一切地想用一些更有生产力的东西来填补空闲时间，于是开始在业余时间开发 API。现在，一年过去了，我有了一些从事好项目和非常落后的项目的经验，我想分享一些我收集到的见解和想法。

# 什么是 API？

最简单的形式是，应用程序编程接口(API)是一种接收一些请求并以一些响应进行回复的服务。通常由开发人员使用，API 是项目将某些元素外包给第三方服务的有效方式。请允许我举一个简单但可能很糟糕的真实例子来说明我现在是如何考虑 API 的:

我们(用户)在酒吧，我们与酒保(API)交谈，以便提出各种请求:

*   你有什么啤酒？
*   我能喝杯啤酒吗？
*   今晚有空桌子吗？
*   我能预订一张下周的桌子吗？

酒保为这些请求提供了一个接口，然后他走开去做工作，最后，你会得到一个回应，比如你的啤酒，一张桌子的预订，或者“我想你今晚已经喝得够多了，伙计”。

重要的是，顾客不需要学习如何倒一品脱完美的啤酒，也不需要在预订系统中搜寻一张桌子。这都是由 API 处理的。酒保消除了我去酒吧的许多麻烦，因此，他们提供的优质服务得到了金钱上的回报。

# API 成功的原因是什么？

## 价值

从上面的例子可以明显看出，客户愿意为给他们带来价值的服务付费。API 的用户通常是那些试图完成他们自己的更大项目的人。这里的价值往往是通过回答这个问题找到的，我能帮上什么忙吗？

作为一个例子，让我们想象一下，有人正试图创建一个基于网络的未来旅游公司。用户将能够搜索要做的事情，订票，并计划他们的旅行。你能提供什么帮助？

*   一个搜罗谷歌来寻找当地名胜的 API 怎么样？
*   一个货币转换器，让用户可以看到他们喜欢的格式的价格？
*   一个 API，可以安排发送短信，提醒用户他们即将到来的旅程？

仅举这个例子，您可以提供数百种潜在有用的服务来为客户的项目增值。

> “在淘金热期间，大多数想成为矿工的人都亏了钱，但是卖给他们镐、铲、帐篷和牛仔裤的人却赚了一大笔”——彼得·林奇

## 规模

像所有伟大的想法一样，最好的 API 往往是最简单的。一个 API 不需要解决成千上万的复杂问题就能成功。事实上，我认为范围失控是失败的原因。相反，关键是专注于为某人而不是所有人提供价值。

# API 货币化

有了正确的想法和目标受众，一旦 API 发布，当然有可能赚到一些钱。有两个潜在的途径可以让你的服务货币化。

**原料药市场。**

API 市场消除了公开发布 API 所需的许多额外麻烦，并开始从中赚钱。从技术角度来看，这是身份验证、速率限制和交易处理，如果实施不当，可能会非常耗时且成本高昂。RapidAPI 这样的市场还可以为您的 API 提供免费公开的额外好处。根据你的服务，你的目标受众可能已经在使用这些市场，因此不需要担心搜索引擎优化和其他形式的营销。

当然，所有好的事情都有不好的一面，这些市场也不例外。首先，这项服务不是没有成本的，你可能会损失一定比例的收入，对于 RapidAPI 这样的平台来说通常是 20%左右。经验丰富的 API 资深人士也将警告过于依赖任何第三方的危险。这些平台遭受的任何中断或停机将最终影响您的客户对您的 API 的访问。

**单飞**

如果你热衷于单干，你可以通过从头开始实现一切来完全避免任何不必要的开支。这有明显的好处，可以降低成本，减少你对他人的依赖。已经拥有坚实用户基础和品牌的现有服务通常会走这条路。对于这些个人来说，在他们现有的产品中添加 API 时，从 API 市场获得的曝光率不太可能与他们从传统营销和 SEO 中获得的流量相匹配。
