# 科技债务

作为一名程序员，我喜欢为我正在做的事情寻找现实生活中的例子。这种对债务的类比让我想到了科技债务。科技债务总是在增长。它可以增长得更快或更慢，但它会一直增长，直到我们进行一次重构。我强调了这一点，因为这是程序员中的热门词汇。重构是代码清洁的转折点。这就像周六的公寓清洁日或九月的世界清洁日。如果我们不这样做，我们会淹没在一片混乱中。

我经常听到产品负责人问*“为什么你第一次没有做好？”*。不是吗？为什么我们在这么短的时间内在应用程序中有这么多功能？我们本可以估计功能会构建得更好，但这需要时间。就像你要准备两顿饭的时候。您可以准备一个，清洁您的工作站，然后准备第二个，或者您可以两个都准备，然后清洁。**所以重构意味着清理。**

有时候，在处理一个特性时，需求会发生变化。通常情况下，我们应该思考我们离改变还有多远。想想看——你挖了一个洞，投资者过来对你说*“我想要一个更大的洞】*。没关系，反正你在挖。当你已经给它浇上潮湿的混凝土时，情况就不一样了。你什么都做不了。你必须让它变干，粉碎它，把它取出来——债务——然后你可以挖一个更深的洞。我建议你下次开发某个功能时考虑一下。

# 产品团队

产品团队专注于应用程序的业务部分——您是技术人员。就像你不理解他们的工作一样，他们也不一定理解你的工作。你必须向产品推销为什么科技债务冲刺是不时需要的。我们设法说服我们的产品团队定期进行技术债务冲刺，以及周五进行小的重构。

# 合适的建筑

我不能说一个架构在应用中有多重要。我看过很多文章说你应该从 POC 开始，放弃测试，因为没有人愿意为它们付费等等。等。这在以服务为基础的公司可能行得通，你为客户准备申请，他们批准，他们付钱，然后他们就走了。

幸运的是，我成功地说服了我团队中的所有开发人员，从领域层和有意义的测试开始非常重要。我是一个超级测试爱好者——我在 TDD 工作。我知道总有那么一个时刻，你会想:“感谢测试！”。从那时起，每个开发人员都会编写它们。有一个巨大的利润——你不工作在 bug 上，你不工作在回归上，即使测试人员或客户发现一个 bug，你创建一个测试来覆盖这个案例，它就永远消失了。

而建筑呢？重要的是要有一个，比如画在 T2 上，让每个人都能看到。它有助于重构，你可以很容易地发现值得回来重新思考的地方。成熟的项目总是有架构，很少看到 POC 转移到成熟的项目。有一条线，如果你通过了，科技债务将无法偿还，这时你就退出了。
