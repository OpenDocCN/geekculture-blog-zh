# 为什么我们不再喜欢 Scrum 了？

> 原文：<https://medium.com/geekculture/why-dont-we-like-scrum-anymore-d1f22f87e4b4?source=collection_archive---------7----------------------->

![](img/b294a4a5c336e726a19ac6afe80de8ba.png)

当我进入这个疯狂的行业时，Scrum 还是一个东西。如今，似乎每个人的梦想都走到了尽头；幻灭开始潜入创业公司和企业内部。对开发人员的压迫，对业务的混乱，比曾经治愈的疾病更糟糕的药物，这就是许多 Scrum 团队开始崩溃的原因

在这篇文章中，我们将讨论围绕 Scrum 的常见批评和不信任，使用一个虚构的来回，在其中我将尝试解决我在经验和研究中发现的主要问题。在文章的底部，读者可以找到一些启发我和给我一些思考的参考资料

**免责声明#1:** 到今天为止，当我们谈论“Scrum”的时候，我们实际上是在谈论一个混合了[Scrum 最初的定义](https://www.scrum.org/resources/what-is-scrum)methodology 和[极限编程](https://www.agilealliance.org/glossary/xp/)的东西:例如，人们普遍认为 Scrum 涉及使用用户故事和故事点来进行评估，尽管这些概念是 XP 实践的一部分。这只是 Scrum 的实现之一，但是为了简单起见，我们假设这是我们打算应用 Scrum 的方式

**免责声明#2:** 经常假设 Scrum 是敏捷的，或者通过使用 Scrum 可以实现敏捷。没有比这更错误的了:敏捷是一种驱动公司向其目标前进的思维模式，而 Scrum 是一种项目管理框架(通常适合软件项目),受与敏捷相同的原则启发。事实证明*敏捷*是帮助创建成功 Scrum 团队的先决条件。

# 作为 MVP 的一切

> 我真的很喜欢创建增量变化和持续评估我们的优先级的想法，但这不是我公司的工作方式:利益相关者希望 MVP 在特定的期限内做出评估并尊重业务计划

让我们回忆一下，MVP 代表最小的产品。尽管有时这个术语并没有按照 MVP 的最初意图来使用，但在实践中，我们确实期望我们正在开发的“产品”是

*   可行:利用给定的资源(时间-人员-金钱)创建一个工作版本是可能的
*   **最小:**只包含增加足够的商业价值来完成 MVP 目标的特征

MVP 的定义似乎违背了 Scrum 的工作方式，但事实并非如此:每个 Scrum 冲刺都可以以 MVP 结束，但在某些情况下，MVP 变得如此“最小”和“可行”,以至于将自己转变为介于敏捷和瀑布之间的怪异的混合怪物

作为一名工程师，很多时候我都有这样的对话

*< <嘿胭脂红！那么，你已经研究了需求，你知道代码和人，你认为你们需要多长时间来实现所有这些东西？> >*

嗯，老实说，我发现了很多极限情况，其中一些在 reqs 中没有提到，而且我们需要一些重构和精确的回归测试。我认为这个项目相当大，如果没有人生病的话，将需要 6-10 周的时间

*< <哦，那是个非常坏的消息人！导演们希望在月底看到这一点，因为他们想在 FooBarEvent 中推出新功能，你知道全世界都将在那里观看我们> >*

Scrum 工作模型背后的精确想法是以这次对话为起点，并将其应用于 MVP，以便(重新)定义范围。[我们不能增加资源](https://www.goodreads.com/book/show/13629.The_Mythical_Man_Month)不要花费比预期更多的时间，这对公司的成功非常重要。所以剩下唯一要做的就是讨论范围

但是怎么做呢？估计总是猜测，总是有忽略重要案例的风险，或者只是在演示的前一天结束为最关键的功能编写代码(*有一天我会讲述我是如何结束在汽车中实现一个功能的，在* [*博洛尼亚-米兰*](https://en.wikipedia.org/wiki/Autostrada_A1_(Italy)) *高速公路上行驶，但那是另一个故事*)。Scrum 方法论的应用已经克服了这些问题。[莫斯科](https://en.wikipedia.org/wiki/MoSCoW_method)方法可用于评估计划活动对产品最终(实际上是初始)提案的影响。**不过，这需要由一名拥有足够自主权和参与权的项目经理/采购专员精心设计，以便与团队一起做出明智的决策。** [**代理产品负责人**](https://www.scrum.org/resources/blog/what-proxy-product-owner-why-it-found-so-often) **是许多组织中发现的一种反模式，在这些组织中 Scrum 方式没有被完全理解或接受**

# 交付物不可发布

> 实话实说:在理想的情况下，做一些小的用户故事是非常好的，让我们的客户看到我们的平台每天都在改进，但是大多数时候，我们将功能分成用户故事的方式是没有意义的！在电子商务部分实现一个“结帐”按钮而没有“清除购物车”按钮有什么意义？

关于用户故事和 sprint 有一个常见的误解，它起源于这样一个想法，即在 sprint 结束时，我们必须释放一些价值，而用户故事应该代表价值的最小增量。所以你可能会想“它真正有价值的是什么？我们是否只关心我们用一个故事引入的边际价值？还是应该有一些因素之间的相互作用？”
既不是前者也不是后者。我们可以通过将*交付*和*发布的概念分开来简化这个问题。*让我们提供一些定义:

*   **可交付成果:**当一项活动满足通过“完成的定义”设定的期望时，该活动就产生了可交付成果。特性的交付是一个战术问题，技术团队负责确保它。例如，我们**可以**考虑一个*故事*作为最小可交付特性
*   **可发布:**当一个特性为产品价值提供了足够的增量，客户准备开始使用它时，这个特性就是可发布的。因此，一个特性的发布是一个战略问题。例如，我们**可以**将*史诗*视为最小可发布活动

因此，很明显，在产品 Backlog 中，我们正在创建一个双层规划，其目标是定义一个包含战术和战略观点的路线图。事实证明，如果没有正确的工具来确保交付和发布之间的有效分离，我们就不能正确地应用这种机制。**技术行业实践是必经之路的基础:功能切换系统、DevOps 文化、SRE 方法论以及所有相关和替代方法**(以及发布策略、实验)是你将使用的工具，将你最近完成的项目从待办事项变为实际生产

# 为什么这是压倒性的？

> 哦，我喜欢 Scrum，我理解它给我们带来的好处，但是我们有计划会议，然后是演示…还有无限的培训会议呢？不要忘记无休止的日常站立，然后复古..还剩下多少工作时间？为什么我不能即时分配任务？

有些人把 Scrum 当成一种宗教，有些人把 whatever 当成一种宗教，也许我们只需要相信一些东西来关闭我们的大脑，并相信我们的行为是最好的，会带我们去天堂。

但是，如果 Scrum 不是一种宗教，我们也不遵循圣经，那么 Scrum 到底是什么？如何确保你的 Scrum 理念在解决引用中提到的问题时做了 Scrum 应该做的事情？

首先也是最重要的:问题不在于你开了太多的会或者开了太长时间，问题在于你没有改进你的流程。Scrum 大师就是为此而生的:**他们不应该是“伪装”的项目经理，而是那些目标只是分析日常工作中的问题并提出补救措施的人，观察某种团队级别的** [**OKRs**](https://en.wikipedia.org/wiki/OKR)

规划会议花了两个小时，因为人们对故事描述讨论得太多了？延长培训会议，与 POs 建立 1:1 的通话来预先细化待办事项，干掉在讨论中过多谈论代码的开发人员，做任何你需要做的事情，但是要记住，大多数敏捷相关方法背后的想法是，这些方法不是成功的秘诀，而是需要培训和细化的形式，以满足必要的目标…这些目标是

*   消除不确定性:每个人都确切地知道它是什么，他们对此知道得如此之多，以至于解决方案是微不足道的
*   **提供可见性:**每个人都知道团队中的所有事情，如果一个人有问题，另一个人会有解决方案
*   **为目标而努力:**最终结果必须是实实在在的，推动团队实现目标的动机是明确的。就技术活动而言，业务方面也是如此
*   **跟踪进度:**团队每天都确切地知道要达到目标还需要什么，并且能够预测目标是否能达到。每个人都应该对团队的整体表现是否有所改善有所感觉
*   **启用反馈:**每一个行动都应该有一个结果，人们有空间和信心提供透明的反馈，并使用它来改进产品和流程
*   **自我组织:**团队拥有外部提供的所有必要工具来实现目标，他们有权不受干扰地做出决策

如你所见，Scrum 不是关于会议，而是这些会议的目的。以最喜欢的团队方式塑造它，以达到这些目标

# 效率还是效果？

> 从我开始使用 Scrum 开始，唯一重要的事情就是完成工作。所有的团队都专注于在冲刺阶段结束时交付故事，而技术债务增加了，我们从来没有时间真正做一些创新

你可以看到，Scrum 是一种遵循敏捷原则的方法论。在这里回忆敏捷是什么以及它培养了什么原则是没有用的:丑陋、古老(但却是金子)[敏捷宣言网站](https://agilemanifesto.org/)对我们来说已经足够了。我更喜欢的原则之一，也是我发现完全适合 Scrum 过程的原则是

> 过程和工具之上的个人和交互

这就是一切:项目的成功是业务需求、技术关注和协作之间平衡的结果，两端都必须定义路线图。只有伟大的专业人士才会创造出伟大的 Scrum 团队，这是一个悲哀的结论，我想除此之外我们别无选择*: **是时候承认每一个承诺让产品看起来比设计它的人更好的解决方案都是一个错误的承诺**

那么我们如何利用这种顿悟呢？核心是谈判、信任和相互理解。业务需求并不比技术问题更重要，它们只是有更快的回报周期。反过来，技术优势是业务需求的结果，因为解决方案只有在解决了某人的问题时才是好的。

Scrum 团队应该有很高的速度，而不是很高的速度。他们必须知道什么时候清扫地毯下，什么时候面对问题。两周的冲刺并不总是以合适的方式完成一个功能的合适时间。总应该有一些改进的空间，比如解决技术债务，做实验。所有这些活动都是过程中的一部分，并为涉众提供价值增量:因为团队本身就是涉众，我错了吗？综上所述，**走向失败的最快途径就是崇拜效率，害怕效果。将您的过程变成没有目的驱动的思维模式的“吃票”,并且无法评估今天的选择的未来影响，将会杀死每一个企业，不管它将采用什么项目管理框架**

**除了作为招聘过程一部分的人员培训或敏捷技能评估；)*

# 结论

最后，我个人认为没有人真的不喜欢 Scrum 或者批评敏捷的目的。有太多的偏差最终把旧的工作方式包装成新的花哨的空盒子，希望能解决问题。这使得人们不再信任这些东西。在过去几年为 Scrum 团队工作期间，我学到的不仅是 Scrum 本身不是一个神奇的公式，而且成功的关键是放弃神奇公式存在的想法。归根结底，敏捷是为了拥抱我们构建的新世界的复杂性，而不是将其隐藏在令人放心的配方后面
