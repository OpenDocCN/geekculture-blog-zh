# 云只是一个工具

## 这不是一个数字化转型计划

![](img/2595b5e4d474660eeae87eb4b382bb6f.png)

数字化转型引起了很多关注，几乎出现在每位首席执行官和首席信息官的年度待办事项清单上。但对大多数公司来说，数字化转型并不容易。也许那是因为我们一直在关注错误的事情。

D 数字变革或颠覆源于成功利用数字技术转变商业模式。新技术和现有技术的创新使用会影响一个行业提供的现有产品和服务的价值。使用术语*颠覆*是因为这些新产品/服务/业务可以颠覆和重组市场。

我们面临的紧迫业务挑战是在不被 it 淹没的情况下控制不断上升的颠覆浪潮。对于初创公司和老牌公司来说，这种挑战可能有些不同，但最终它会比其他人更具创新性、更有效、更敏捷。成为颠覆者确实很好，但不能有效地应对颠覆可能是致命的。云在其中扮演什么角色呢？

## 云

“云”这个术语最初是作为技术术语出现的。在互联网的早期，技术图表通常用云形符号表示互联网的服务器和网络基础设施。随着越来越多的计算机处理和数据转移到云中，这个术语成为互联网/云计算本身的简称。

如今，我们可以在公共云、私有云或混合云上运行我们可以在内部企业数据中心运行的几乎任何东西，作为可租赁、可联网、可动态扩展的资源，我们可以将几乎任何可联网的设备(电话、平板电脑、笔记本电脑、PC、机器、物联网)连接到这些资源，而容器和容器编排可以使我们更容易地以越来越小的增量构建、部署和管理应用程序功能。那可以改变一切。但是云仍然只是一个工具。它是运行和连接软件的地方，而不是软件应用程序本身。我们仍然需要建造、租赁或购买这些设备。

## 优先

问题是如何确定我们的优先事项。我们的数字颠覆计划的第一步是什么？我们是先迁移到云，还是先实施转型应用程序，还是尝试两者同时进行？明智地选择，因为这里有龙。

让我们从客户的角度来看一下我们的优先事项。我们的客户更有可能购买我们的产品或服务，是因为它让他的生活或业务比我们竞争对手的产品或服务更好，还是因为我们将它部署在云中？他会知道它是否部署在云中吗？从客户的角度来看，云真的重要吗？如果没有，那么…

## 时间和资源

归结起来就是时间和资源。尽早满足客户的需求是一件好事。如果是竞争对手在追赶，那么与竞争对手进行追赶会有趣得多。因此，让转型应用程序更快地与客户取得联系肯定会更好。

资源完全是另一回事。很少有公司拥有无限的 IT 和开发资源，拥有过多云技术能力的公司就更少了。我们中的许多人没有资源来同时处理大型云迁移和重要的应用程序开发。我们需要选择云迁移或新应用程序开发，很难两者兼顾。

***让我们明确一点，云技术就是未来。我们不能忽视它。虽然我们可能需要在解决云迁移之前解决应用程序开发需求，但是我们必须以一种方式来完成，以便在我们准备好的时候能够将我们的新应用程序部署到云上。幸运的是，有一种方法可以做到这一点，而无需花费更多的时间和资源。***

## 前进的道路

我们能否在内部计算资源上构建和部署新的云原生应用，并推迟全面迁移到云？答案是响亮的“是！”—这比你想象的要容易。关键促成因素是:

1.  像 Docker 和 Kubernetes 一样使用 [**容器**](https://www.docker.com/resources/what-container) 和 [**容器编排**](/nerd-for-tech/building-cloud-native-apps-part-1-393d250cf107#8f4e) 。这些技术是在云上打包、部署和扩展应用程序组件的首选方法。Docker 和 Kubernetes 都可以用于内部数据中心，甚至可以在 PC(Linux、Mac、Windows)上进行开发和测试。如果容器运行在本地计算机或 PC 上，它可以在适当配置的云供应商集群上不加修改地运行。
2.  使用 [**微服务架构模式**](/@dick-dowdell/designing-microservices-4130bc41c046) 交付开发云的自动化部署、故障转移和扩展功能所需的组件大小和灵活性。微服务架构模式还优化了敏捷软件开发、交付和维护实践。
3.  使用 [***扼杀者图*模式**](/nerd-for-tech/building-cloud-native-apps-part-1-393d250cf107#62a5) 来简化从遗留协议和 API 到最终微服务替代品的过渡。
4.  避免使用内部不可用的专有云供应商技术，这些技术会将一个供应商局限于特定的供应商。有些应用需求可能只有通过 AWS、GCP 或 Azure 才能得到更好的支持。如果这是绝对必要的，那将会把一个人锁定在一个特定的供应商身上，并加速迁移到云的需求。

## 包扎

以利用云架构优势的方式构建我们的新应用，但推迟向云本身的实际迁移，具有一些真正的优势:

1.  保持对关键应用需求的关注和投资，这些需求对我们业务的竞争态势有最积极的影响。
2.  优化我们利用敏捷方法进行更快速开发和特性迭代的能力。
3.  充分利用我们有限的开发和 IT 资源。
4.  积累设计、实施、部署和管理云原生应用程序的经验，最初不会带来迁移风险。
5.  让我们做好准备，在迁移时机到来时，利用知识和经验更好地管理云迁移的风险。

云技术是未来。我们不能忽视它。但是我们可以管理我们自己的过渡。
