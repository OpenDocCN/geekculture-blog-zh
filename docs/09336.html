为什么没人看你的软件工程简历

原文：https://medium.com/geekculture/this-is-why-nobody-is-reading-your-software-engineering-resume-49d8c695ea25?source=collection_archive---------5-----------------------#2021-12-06

你可能是一个经验丰富的优秀开发人员。也许你是团队的核心贡献者，也许你是团队或产品的领导者之一。但是..你的简历？它会告诉陌生人你唯一擅长的是什么吗？还是仅仅告诉他们你使用了什么技术，就像其他java开发人员一样？

如果它像我们看到的大多数开发人员简历一样，它可能是这样的:

  • 利用Spring Boot、Hibernate、JPA和GraphQL SPQR开发后端GraphQL API
  • 企业产品配置软件的全栈开发
  • 创建并实施了多个spring boot微服务

我对这些开发人员了解多少？他们知道java和spring boot。也许这很有帮助，但是作为一名招聘经理，我的招聘人员已经过滤了我的一堆简历——几乎我名单上的每个开发人员都知道java和spring boot。更不用说，我们都遇到过了解java和spring boot，但实际上并不那么好的开发人员。

那么，他们应该说什么呢？

为了证明你的价值，你需要清楚地说明你对工作的三个关键结果的影响:用户影响、系统适应性和团队流程。

你的影响的三个结果

1.用户和业务影响

你开发的技术以某种方式影响了用户。告诉我们那个用户的情况！他们是谁，有多少人，你是如何改变他们的生活的？你造的那个东西为什么跟有关系？它对谁有价值？

将多页用户体验迁移到单页应用中，将客户参与度提高了8%

React中为一项新功能设计的产品之旅将客户采用率提高了22%,带来了110万美元的增量收入

2.系统适应性

讨论你开发的功能很简单。但是我们经常忽略讨论我们必须集成到项目中的非功能复杂性。你的产品最具挑战性的方面是什么？从建筑的角度来看，你是如何做到的？系统需要如何发展，你在其中扮演了什么角色？

构建了RESTful APIs，在整个公司使用，为超过250，000个并发用户的React前端提供数据

制定并实施了对清理流程和性能的改进，将停机时间减少了90%

3.团队流程

如果我们在工作时只需要考虑代码——这可能会非常简单，对吗？技术组织成功的一个关键因素是他们的团队，他们如何一起工作，以及在交付生命周期中想法的流动是什么样的。您如何影响生命周期和团队是至关重要的！你是如何简化开发人员的工作流程的？你为他们解决了哪些问题？你是如何改进你的生产之路的？可以说说省了多少时间！

重新构建了我们的平台方法，使团队能够将部署频率提高400%

为内部UI库做出贡献，以创建可重用的组件，每月节省200小时的部署时间

简历是你的第一印象，也是你进入职场的第一步。写得好的简历会带来更高的薪水，更多的高级职位，以及更短的求职时间。通过谈论你对每个项目的切实影响，确保你的项目高于其他项目。
