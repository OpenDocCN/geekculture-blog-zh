# 潜在的软件开发问题应该使用多种因素来识别

> 原文：<https://medium.com/geekculture/underlying-software-development-problems-should-be-identified-using-multiple-factors-d041a63b0b76?source=collection_archive---------15----------------------->

![](img/0ac75ec892a44516f6df4bb7a3875795.png)

一个组织解决一个复杂的软件开发问题，使用同样的[技术](/geekculture/an-organization-must-manage-complicated-difficulties-in-software-engineering-the-way-it-handles-f0db94ecf80f)来解决其他领域的复杂困难。这些方法可以用来纠正复杂的[系统问题](/geekculture/systemic-problems-in-software-development-are-as-complex-as-they-are-everywhere-else-60cc56e8500f)。然而，一个机构可能无法通过快速检查来确定某个特定的头痛是一个孤立的单元还是系统的一部分。如果企业投入时间和资源去挖掘一个根本性的失败，它就冒着一场徒劳无功的巨大支出的风险。为了避免陷入焦油坑，它对挖掘系统性问题持谨慎态度。它可能会更加犹豫是否要寻找潜在的软件开发问题，因为它已经在该领域投入了大量资源。然而，应用程序的构建受到适用于其他领域的相同的[一般规则](/geekculture/the-generally-relevant-rules-apply-to-programming-too-e780a54daf68)的约束。这些法律有助于组织确定是否存在根深蒂固的问题。如果它使用这些指导方针，它就减少了参与鹬狩猎的可能性。为了避免徒劳无功，它应该运用这些方法。软件开发中的一个基本问题看起来类似于一个机构，类似于其他部门的机构。

编程之外的问题集中在进度、资源分配和实现目标上。如果一个项目错过了它的截止日期，超出了它的预算，或者没有达到它的目标，那么它就是一个[失败](/geekculture/software-development-projects-fail-for-the-same-reasons-ones-in-other-domains-do-3bea91995be1)。失败是一个组织想要避免的。一家机构尤其希望避开频繁的亏损。它高度关注导致这些失误的系统性困难。如果一个问题导致一项努力错过了它的截止日期，那么一个公司很可能对此感兴趣。那家企业可能会赔钱或失去尊重。如果一个组织经历了足够多的损失，那么它可能会从这些属性中失去很多。频繁的失败代价高昂，所以他们引起了一个机构的极大兴趣。如果一家公司确定这些失误是潜在问题的结果，它就会高度关注这个问题。软件开发中的困难包括调查进度、预算和目标，就像他们在其他领域做的一样。应用程序构建中的一个基本障碍对一个企业有很大的吸引力，它看起来非常类似于不同领域中的问题。

不管是哪个部门，根本的复杂因素都不是一个组织可以通过简单的方法识别的。例如，一个机构可能注意到它的项目以一定的比率失败。然而，这一频率本身不足以确定一家公司是否存在初级问题，除非该百分比非常高(例如，95%)。企业需要权衡多种因素，以确定是否存在潜在的困难。例如，它需要知道自己的努力在多大程度上未能对其前景有一个更清晰的了解。如果它的承诺经常失败，但只是在小的方面，那么它的方法可能没有问题。然而，如果它的项目很少失败，但失败的方式很严重，那么它可能会有问题。一个组织需要多种属性来正确识别系统性困难。

其中一个方面是规律性。随着一家机构的失败率上升，它出现根本性问题的可能性也会增加。如果该企业从事软件开发，高密度的失败也是一个很好的指标，表明它在该领域有一个基本的问题。例如，公司的努力错过了他们的时间表，超出了他们的预算，和/或经常达不到他们的目标。该组织可能在应用程序构建方面存在系统性困难。编程可能具有高度的不确定性，使其任务难以准确估计。任何做出超出确定性水平的承诺的努力都有失败的风险。如果一个机构定期做出这样的承诺，它的项目将会不断失败。组织必须考虑频率，以确定它是否有系统的软件开发问题。

然而，除了极端情况之外，一个公司仅仅使用重复是无法得出根本缺陷的。一个机构必须考虑其失败的程度和频率，以确定它是否有潜在的问题。一家企业不太可能出现问题，只要有足够的规律性，就能让其他因素变得无关紧要。该机构必须考虑其失败的程度，以获得更清晰的画面。例如，一个组织可能有一半的项目失败了，但都是轻微的。另一家企业可能会有十分之一的项目倒闭，但规模巨大。第一家公司可能没有系统性问题，而第二家公司有。如果那个机构做软件开发，那么它失败的程度与它的承诺和它对承诺的确定性之间的差异有关。如果 it 部门对项目的某个方面不确定，并且围绕该要素做出了很大的承诺，那么就该努力的时间表、预算和/或目标而言，它很可能会遭受重大挫折。如果一个编程企业过早地承担义务，那么它必须参与返工，这会影响时间、资源和实现目标的能力。这些修订的规模与承诺的确定性和承诺之间的差距成正比。随着编辑变得越来越大，组织失败的程度也在增加。一个机构必须考虑其失败的程度和频率，以确定它是否有系统性的应用程序构建问题。

该公司必须检查的不仅仅是重复和规模，以判断它是否存在根本问题。原因的一致性是企业是否存在根本性问题的一个因素。它的失败可能规模不大，也不太常见，但它们仍可能是统一缺陷造成的。如果一个组织的失败有相当一部分是由一系列一致的原因造成的，那么它就有一个潜在的问题，即使它的损失不经常发生，也不严重。如果这些障碍是在软件开发中，那么这个机构仍然有一个基本的问题。应用程序的构造可能涉及到很大的模糊性。一些公司没有考虑到这种差异，导致在同一领域(如项目规划)的重复失败。他们的努力可能不会经常失败或者失败得很惨，但是因为相关的原因。原因的一致性是一个企业是否有系统软件开发问题的一个因素。

该组织还必须考虑其失败原因的不可预测性。如果它猜不出失败的潜在原因，那么它会收到一个信号，表明存在一个根本性的问题。一个基本的障碍既有秩序，也有混乱。如果这种斗争涉及到应用程序的构建，那么一个机构很可能无法处理其项目中的可变性。每一次步履蹒跚的努力都以不同的方式表明了该公司的失误。例如，一个企业可能正确地确定了企业的范围和时间表，但是没有分配足够的资源来满足该时间表。然而，在不同的情况下，它可能很容易正确地确定企业的广度和资源，同时错误地判断该努力的日程表。编程中的许多努力都涉及不确定性。如果这个方面是一个组织的基本斗争，那么它的失败将以不可预测的方式暴露出来。根本原因的混乱是系统性软件开发困难的一个因素。

然而，一个潜在的问题有许多方面。它不能由一个机构通过简单的手段来识别。一个单一的因素可以确定一个基本的斗争，但只有当它位于极端的边缘。在这个界限内，公司应该使用权衡多种因素的方法。一个企业可能需要权衡混乱、可预测性、失败的频率和失败的程度，除非其中任何一个是边缘情况。本文不探讨组织在软件开发中应该考虑它们的方式。这篇文章没有描述一家机构如果认为自己存在系统性问题，应该怎么做。这些主题将在今后的工作中讨论。
