敏捷曾经是将我们从繁琐的瀑布式文档中解放出来的伟大解放者，现在已经成为大多数技术组织的必备要素。

尽管标题尖锐，这篇文章将公平地看待敏捷 scrum 和看板——它在数据科学中的潜在应用，以及为什么你可能想或不想在你的商店中利用它。如果你选择使用它，你会学到一些技巧，避免一些陷阱，让你和你的团队成功。

“数据科学”在这里将专注于训练、测试和比较模型的功能，无论它们是统计还是机器学习。我们将假设对敏捷实践有一些熟悉，并且使用像“冲刺”和“冲刺规划”这样没有定义的术语。如果你是一名数据科学家或产品专家，正在努力使敏捷成为数据科学团队的一项有价值的实践，这篇文章将会有所帮助。

## 你应该对数据科学使用敏捷吗？

当然，如果你致力于清晰地观察数据科学的工作是如何完成的，并以一种深思熟虑的方式使用敏捷来支持这项工作。

> 在我们的模型移动目标指针之前，我们工作的价值是假设的。

如果我们暂时忘记寻租型敏捷顾问的家庭手工业以及由此产生的艰苦过程，回到最初的[敏捷宣言原则](https://agilemanifesto.org/principles.html)，我们在机器学习中会发现很多共鸣。当然，我们中的许多人会把这些原则中的几个看做是过分关注软件，例如，“工作软件是进步的主要衡量标准。”我们机器学习专家有一种冲动，要将这一原则更多地推向模型结果或度量。在我们的模型移动目标指针之前，我们工作的价值是假设的。(稍后我们将讨论可以轻松合并这些视角的实践。)“围绕有动力的个人建立项目。给他们需要的环境和支持，相信他们能完成工作[，]”是一个真正超越学科的原则。

> “围绕有积极性的个人建立项目。给他们需要的环境和支持，相信他们能完成工作。”— [敏捷宣言](https://agilemanifesto.org/principles.html)

当然，现实世界中的组织很少真正从这些原则中获取动力。以我的经验来看，更常见的是敏捷的商店，例如，“我们的产品负责人喜欢 scrum”，或者“整个公司已经在吉拉了。”这是一个许多开发人员都有的幻灭时刻:当她意识到现代敏捷实践更多地关注于跟踪和报告一个组织不信任的开发人员的活动，而不是关注为追求卓越带来快乐。

跟踪或报告本身没有问题，相反，它们对组织的所有部分都非常有价值。至关重要的是，一个组织要确保在计算了工作的跟踪、优先级排序和沟通的成本后，这些活动能获得正的投资回报。如果做得好，这种回报可能是巨大的，以降低团队和更广泛的组织的沟通成本的形式。也就是说，你花在告诉别人你在解决什么问题上的时间越少，你花在解决问题上的时间就越多。对于许多数据科学家来说，集中精力解决难题所带来的高效感是这份工作的真正乐趣。

## 数据科学和机器学习有什么特别之处？

不确定性。数据科学家的主要角色是增长组织的知识，通常是通过推理统计，我们量化不确定性并应用统计测试；或者，通过构造证明，我们证明机器学习可以通过构造一个解决问题的黑盒来解决给定的问题。知识的生产和这个过程是不确定的。

> 数据科学家的主要角色是增长组织的知识。

这可能会在跨学科团队的不同职能之间产生一些摩擦。产品专家主要关心的是比竞争对手更快地将产品推向市场，这是正确的。面对一个新的机器学习功能，不习惯使用机器学习的产品经理经常会问一个看似无害的问题，“这个模型什么时候可以部署？”毕竟，寻求依赖关系的清晰性并绘制路线图是他们的工作。一个善意的数据科学家通常会给出事实上真实但组织上幼稚的回答，“没有办法知道。”

> 我们必须将我们对交付 DS 单元的概念从建模结果转变为可复制知识的**工件**。

这对习惯于在敏捷商店工作的人来说可能是不和谐的，这些天是最不和谐的。sprint 周期的前提是调整你的工作承诺，因为你越来越了解一个团队在这个周期中可以交付什么。估计完成时间对于软件 IC 和领导来说是一项关键技能，所以当 PM 听到句号“不知道也不可能知道”时，他们会根据经验变得有点怀疑。“我们以前没有构建过如此复杂的特性，所以我们的估计可能会有些偏差”是敏捷开发人员的自然说法；“无从得知”对很多经前综合症患者来说就像无能一样。

这种频繁的微沟通错误并不是不可克服的，它只是不同学科之间不同思维模式的产物。为了克服它，我们必须将我们对交付的 DS 单元的概念从建模结果转移到可再生知识的**工件**。

## 我们如何使用敏捷来成功地管理这种不确定性？

如果你是一个数据科学家，在敏捷框架中协商工作单元，只承诺可估计的建模工作单元；永远不要承诺对结果建模。如果你是项目经理，永远不要在你的故事、任务和接受标准中寻求承诺**来塑造结果。请注意，这一原则适用于**承诺可交付成果的具体单位**的层次，而不是项目关键目标的层次。**

> 科学是一个过程:假设、实验、测量、重复。您提交的项目任务应该每个都在这个周期中捕获一个迭代。

也许有一个魔鬼在你的肩膀上低声说，“好吧，永远不要说永远……”请把那个声音驱逐到另一个房间:那个声音是我们不能拥有美好事物的原因。

通过“建模结果”，我指的是模型性能的可测量的改进。对于一个对建模结果的承诺的例子，以接受标准为例，“模型更新将改进我们的转换模型的相对误差 2%。”像这样的创作标准是一个错误。它假设存在一种影响，并且这种影响具有显著的重要性。科学是一个过程:假设、实验、测量、重复。您提交的项目任务应该每个都在这个周期中捕获一个迭代。

如果你是一名数据科学家，把你的票集中在验收标准上，这些标准完全在你的机构内，在分配的时间内交付，并且可以被观察到是无可争议地完成的。例如，对比这两个标准:

1.  模型培训渠道将包括三个新的、特定的预测功能。将报告使用和不使用新特征训练的模型的预测性能的差异。
2.  模型培训渠道将包括新功能。特征将改善模型性能，通过均方误差减少不低于其当前值的百分之三来测量。

在(1)中，我们有一个固定范围的工作单元，就数据科学家的时间而言，它的输入可以像典型的敏捷软件商店中的任何其他技术任务一样进行估计。(ML 新手可能会对这里的术语“预测特征”感到困惑。这是 ML 社区版本的“独立变量”，认为“输入到我们的模型。”)就像任何其他敏捷商店一样，团队的承诺可能需要调整。随着数据科学家在团队数据基础设施和工作流方面的经验增长，她将能够在一次冲刺中添加更多的特性。如果组织转移到一个可发现的特性库，这种工作将可能达到真正的速度。所有这些将使一个组织能够学习更多关于特性提升的知识，而不提供哪怕是一丁点儿的保证，来保证我们将会学到什么或者结果会是什么。

在(2)中，我们致力于一项持续时间介于一次冲刺(如果我们非常幸运的话)和从现在到宇宙热寂之间的任务。

回想一下我们的代理和可观察状态的激励原则。目标提升是可以观察到的，但是为了在一次冲刺中实现这个提升，还有什么是真的呢？

*   组织必须拥有基础架构，使数据科学家能够检索新功能并比较使用和不使用新功能时的模型性能，并且数据科学家必须具备足够的技能在规定的时间内完成这些工作。
*   这些特征必须在现有模型的假设空间内提供提升。

首先是团队应该能够从经验中学习的东西。第二个条件可能是真的，回答它是否是真的是添加特性和执行模型比较实验的全部意义。

> 科学的工作是回答问题，我们永远无法保证答案会是什么:我们所能承诺的是我们会努力找到答案。

精明的读者会注意到，科学方法包括一个预测步骤，在这个步骤中，我们从被测试的知识和理论中得出一个可证伪的预测。有人可能会问，把这种预测写进故事或彩票有什么错？

本质上，什么都没有。为了成功地将敏捷用于数据科学，组织必须理解**期望**和**承诺**之间的区别。“根据以前类似型号和功能的经验，如果我们看到 3%的提升，我不会感到惊讶，”表达了一个示例预期，这是在 sprint 规划或票证评论中说的明智和理性的事情。但是科学的工作是回答问题，我们永远不能保证答案会是什么:我们所能承诺的是我们会努力找到答案。

# scrum 和看板哪个更适合 DS？

既然我们已经将数据科学的任务重新定义为创建知识单元，我们已经为敏捷中成功的 DS 奠定了基础。人们自然想知道 scrum 和看板风格的项目管理之间是否有很大的区别。总的来说，scrum 以固定长度的冲刺来计划工作单元，团队可以选择在这段时间内要做多少工作。看板是一种连续的流程模型，旨在最大限度地减少作业，最大限度地提高生产能力。(更多信息，来自 Atlassian 的这个[帖子是一个有用的、全面的介绍。)简单的回答是这两种方法都可以，但是这些有针对性的问题可以帮助你的团队找出最适合的方法:](https://www.atlassian.com/agile/kanban/kanban-vs-scrum)

*   其中的一个已经在您的组织中广泛使用了吗？那是用制服吗？每个人都有共同的两周冲刺节奏吗？
*   您的 MLOps 和从原型到生产的部署实践有多完善？(例如，我已经在这里[写了关于 Make 如何减轻对工程界朋友的负担的文章。)](https://towardsdatascience.com/using-make-for-data-science-projects-changed-my-life-23de1597e2e)
*   你的团队更关注通过部署的机器学习的价值，还是通过告知策略的推理模型的价值？

从回答每一个开始，然后考虑他们如何一起玩。例如，假设您的开发团队使用共享的 scrum 步调，您的团队专注于通过大数据模型了解客户。如果你的角色是通知产品积压决策，你应该尽可能快地学习所有你能学的东西，有人可能会说看板可以帮助你做得更好，而且不会牺牲任何东西。

相反，如果您依赖开发团队来构建服务于您的模型预测的微服务，您可能会从共享相同的节奏中获得巨大的价值。如果您的 MLOps 实践足够成熟，您能够快速独立地培训、验证和部署模型，并且能够监控数据和模型漂移，独立地重新培训和重新部署，那么这可能会改变一些事情。也许您可以自由地采用不同的步调，或者也许使用您的模型 API 的团队会欣赏共享步调为您提供的路线图的简易性。如果你们有独立的分析团队，也要考虑他们的步调，以及你们如何相互放大彼此的价值。(我已经在这里写了更多关于这个[。)](https://towardsdatascience.com/analytics-and-data-science-are-better-together-f5190a7e91b9)

这在很大程度上是一刀切的情况。但这里有另一个案例，员工的快乐可以指导我们:如果你记得交付的障碍让数据科学家沮丧，而影响带给我们的快乐增加了快乐，你可能会找到正确的答案。

# 总结和开始

首先，一定要意识到做好这件事是一个挑战。同样，不同的不确定性向量导致了与典型软件非常不同的工作，组织必须一起转变他们的思维模式。根据两个阵营的个性，这一挑战可能会更加复杂。有些产品经理从他们对待办事项进行优先级排序的角色中推断出一定的权限，可以将“我负责”的态度带入每个交互中，还有太多的数据科学家将他们的定量天赋视为权限，可以对任何不能立即解释标准化变量的对数优势关系的人进行微攻击。

1.  如果你已经处于一个 scrum 风格的敏捷环境中，并且上面描述的挑战已经在你的环境中显现出来，那么在你的下一次冲刺计划或事后分析中提出它们。提前说明你不能承诺建模结果，只能承诺记录找到的答案的可交付物。
2.  如果您没有使用敏捷，并且想尝试使用它来进行数据科学，那么从最轻量级的管理系统开始吧。如果您的组织已经在 GitHub 上，他们的项目特性可能是启动轻量级看板流程的一个很好的选择。
3.  如果你的利益相关者不知道该为你的待办事项请求什么，考虑给他们一个选项菜单，描述你的团队能交付什么(完整的细节[在这里](https://towardsdatascience.com/to-improve-your-relationships-with-ds-stakeholders-offer-them-a-menu-dfc15f3006c6))。更好的是，开始一场关于在整个组织中嵌入您的数据科学团队的[的对话。](https://towardsdatascience.com/to-make-the-most-of-your-data-scientists-embed-them-f29c3c9ed57d)
