# 为什么每个 SaaS 都需要良好的开发者体验——以及如何提供

> 原文：<https://medium.com/geekculture/why-every-saas-needs-a-good-developer-experience-and-how-to-provide-it-9d2d81467231?source=collection_archive---------15----------------------->

![](img/5ba36bccea9cc3326f2eaa6ed0e401cd.png)

大多数 SaaS 公司不会在他们的开发者体验上投入太多。但是随着产品集成越来越重要，强大的开发者体验不再仅仅是开发者和企业软件的事情。

增长最快的 1000 家 SaaS 公司平均有 15 个产品集成，而前 15 家公司有 347 个集成。由于早期阶段的公司[需要为他们的客户提供开箱即用的集成以达成交易](https://chiefmartec.com/2020/09/integration-important-criteria-selecting-martech/)，公司正在研究如何更快地为他们的客户提供更高质量的集成。

扩展集成产品需要依赖合作伙伴和第三方开发人员来构建、维护和支持集成。

通过为这些合作伙伴开发者提供卓越的体验，公司可以围绕他们的应用建立一个繁荣的生态系统，而不必自己支付和支持每个集成。

但主要用户不是开发商的 SaaS 公司可能没有花太多心思向开发商营销或支持他们。

即使在早期阶段，每个 SaaS 公司都可以为合作开发人员提供良好的体验，如果他们关注他们的旅程和观点的话。

在为他们的生态系统构建开发者体验时，公司应该考虑第三方开发者之旅的 6 个阶段。

# 意识

开发商不想以传统的方式被“推销”。相反，提高开发人员的意识应该围绕事件、黑客马拉松和有益于开发人员职业生活的信息内容。

通过在活动中发言、撰写关于生态系统相关主题的文章、参与 Reddit、Discord、Quora 和 DEV 等社区，或者展示您可以使用 API 做的很酷的事情，让开发人员参与到您的生态系统中，可以帮助第三方开发人员提高知名度。

# 探测

对于生态系统开发者，探索将集中在你的产品和 API 文档上。除非你有令人信服的理由，否则你的 API 文档应该是公开的。开发人员不想花时间在你的网站上挖掘，或者只是为了看文档而请求登录。

你的文档也应该是彻底的、更新的和清晰的。

第三方开发人员不是你的产品甚至你的产品类别的专家，所以一定要不仅解释你的 API 的技术组件，还要给出产品和 API 为最终用户做什么的高层次解释，以便他们能够理解用例。

对于早期公司来说，遵循 OpenAPI 规范可以简化文档流程。像 [Swagger](https://swagger.io/) 或 [Readme](https://readme.com/) 这样的工具可以自动生成交互式文档，这样可以节省时间并确保文档保持最新。

许多公司遵循[文档即代码流程](https://www.writethedocs.org/guide/docs-as-code/)来维护他们的文档，该流程遵循版本控制、代码审查和测试的基本编码流程。

更成熟的公司提供 SDK 和更多的例子，这使得 API 更容易使用。

# 考虑

当开发人员考虑是否为你的应用构建一个集成时，他们会更深入地研究你的 API 和它支持的用例。

请记住，合作伙伴组织通常会在为您的产品或您的竞争对手之一构建集成之间做出选择，这个决定的一个关键部分是您的 API 的易用性。

你应该给开发者提供一个免费的沙盒账户，因为这有助于他们通过实践的方式更好地理解你的产品和 API。理想情况下，开发者只需要输入他们的名字和电子邮件就可以访问。

沙盒应该清楚地记录它们与普通帐户的不同之处。使它们尽可能与常规帐户相似，并包含各种用例的数据。

API 调用限制应该与普通帐户相同，即使超时要快得多。这使得开发人员能够确保他们的集成在真正的限制内工作。

在评估是否值得为您的产品构建一个集成时，您的 API 设计起着很大的作用。

大多数 SaaS 公司提供带有 JSON 响应的 REST APIGraphQL 越来越普遍。大多数开发人员都熟悉这些风格和框架，所以只有在你有充分理由的情况下才偏离。

[OAuth 2](https://oauth.net/2/) 是认证和授权的标准。同样，只有在你有很强的商业理由的情况下，才偏离这一点。

就 API 的资源、端点和调用限制而言，确保尽可能使用通用语言，并考虑不同的用例。

调查不同类型的客户，以了解他们在其他系统之间移动数据时的需求。如果你在一个既定的产品类别中，看看竞争对手的 API 提供了什么。

如果您的客户需要从他们的系统中获得一些实时信息，那么可以考虑提供 [webooks 或流媒体 API](https://www.pandium.com/blogs/what-are-webhooks-an-explanation-for-the-non-technical)，这样合作伙伴就可以高效地设计产品集成，实时获取他们的客户数据。

# 开发和测试

完整的 API 文档和健壮的沙箱在提供开发和测试集成的良好体验方面发挥了重要作用。

允许开发人员将团队成员添加到他们的帐户中，在测试和设计工具(如 Postman)中提供集合，以及启用测试版应用程序也可以改善测试体验。

在这个阶段，技术支持至关重要。作为一个较小的公司，提供一对一的支持可以为您的 API 设计和用例提供宝贵的反馈。

如何联系支持人员以及期望从支持人员那里得到什么应该在开发人员门户中明确，或者如果没有的话，在 API 文档中明确。给出时间表并对可用性持现实态度有助于避免增加挫折感。

如果您不能提供一对一的技术支持，即使是通过电子邮件，也要密切关注支持查询，这样您就可以更新您的知识库甚至您的 API。您还可以考虑尝试在您的生态系统中找出能够帮助同行的开发人员。

# 发布和维护

一旦开发人员完成了对测试版客户的测试，他们会想在你的市场上发布它(如果你有市场的话)或者让你的客户看到它。提供清晰的文档，说明如何在您的市场中发布应用程序，以及技术和营销材料方面的要求。

如果在你的市场中发布需要开发工作(如支付处理)，尽可能简单明了并有良好的文档记录。理想情况下，提供一个门户，开发者可以在那里提交和接收对他们应用的反馈。

在开发者门户中展示开发者的应用和市场分析也让他们更容易支持客户和改进他们的应用。

# 参与和宣传

参与其中的开发人员将更好地支持和扩展他们的集成。他们还会把你的生态系统推荐给其他开发者以及他们在产品和合作关系方面的同事。

奖励和认可那些开发了很酷的东西或者为你的社区做出贡献的开发者可以带来更多的参与。奖励和认可可以包括奖品、博客简介、在活动中发言，或者与您的产品和工程团队取得联系。

征求反馈和组建开发者顾问团也有助于吸引开发者并把他们变成拥护者。通过指定的第三方开发者主机促进本地聚会可以增加投资和社区。

一旦您为开发人员旅程的每个阶段设置了资源，您将希望跟踪指标以确保开发人员对体验做出积极的响应。

在每个阶段实施开发人员调查可以补充统计数据。例如，在你的文档和开发者门户上放置一个“请求特性”按钮。

就像传统的收入漏斗一样，你的目标是确保有足够多的开发者在前进。

您应该跟踪以下开发人员的数量:1)访问您的开发人员专用页面和 API 文档 2)注册您的开发人员门户和沙盒 3)经常使用沙盒 4)启动应用程序 5)提交应用程序以在您的市场上发布 6)维护和支持他们的应用程序 7)培养满意的客户群 8)放弃他们的应用程序 9)成为社区的积极成员。

除了对应用程序的贬低(这是你不希望鼓励的)，任何阶段的严重下降都可以提醒你开发过程中的问题，并表明你需要在哪里投入更多资源。

最有可能的是，你的最初目标将是让更多的开发者推出一个受支持的应用，并培养一个满意的客户群。更成熟的组织可以在整个漏斗上花费更多的资源，包括提高认识和宣传。

开发人员体验不再是只有开发人员和企业产品需要关心的事情。通过投资[引人注目的开发者之旅](https://www.pandium.com/resources/learn-how-to-create-an-exceptional-third-party-developer-experience)，任何 SaaS 公司都可以增加他们提供给客户的高质量集成的数量。反过来，这些整合将有助于达成更多交易，增加客户保留率，这对于成功的增长变得越来越重要。
