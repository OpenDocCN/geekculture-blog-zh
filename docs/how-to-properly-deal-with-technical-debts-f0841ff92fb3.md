技术债务可以被定义为做出一个[几乎]完美的变更(完美的架构设计，使用良好的编程实践和标准，更新文档，彻底测试变更，等等)之间的差距。)并利用一些可用的资源尽快做出改变。

> 请记住，技术债务是一个比喻，而不是一个科学概念。类似于“货币债务”，技术债务不一定是坏事，有时它是推动项目前进所必需的，但是它必须得到正确的管理。

想一个买房的例子。如果你想等有足够的现金购买房产，你需要等很长时间。鉴于目前的资源，你可以获得抵押贷款，现在买房，过一段时间再还清。你最终会付出更多，但也可能有好处，比如房价上涨。

在软件环境中，在特定的时间发布一个特性是有价值的。如果我们意识到了技术债务并讨论了它的利弊，为了获得一些优势，尽早发布软件可能是好的。然而，你应该管理好技术债务，不要让它溜走。以买房为例，在没有计算好经济风险的情况下，就决定先买车再买船，那你就破产了。对于软件开发，你不能忽视债务，因为下一个周期你可能无法有效地发布另一个特性。但是如何确保我们恰当地管理我们的技术债务呢？

# **1。确定你的债务**

没有特定的工具可以在项目上运行来识别其底层的技术债务，但是，一些工具如 smart IDEs 可以在项目代码中标记一些潜在的债务。另一方面，“过时的维基”不是自动化工具可以捕捉到的。以下是一些技术性债务指标的例子，你可以在日常交流中发现:

*   目前不要担心文档！
*   暂时还可以，我们以后再重构。
*   //TODO:这应该在发布之前修复。
*   我知道如果我碰了那个密码，一切都会崩溃。
*   这已经足够好了，我们将在下一个版本中完成测试。

在你目前的职位上，你可能会发现类似上述的陈述，表明可能存在技术债务(请将你的例子添加到评论部分)。

# **2。估计成本和风险**

对于每一笔技术债务，都存在眼前利益和潜在未来成本之间的权衡。在做出最终决定之前，仔细评估债务的潜在风险。“不更新 wiki”乍一看似乎不是一个很大的风险，但是想象一下这样一个场景:您添加了一个关键特性，但是没有花费几个小时来更新剧本 wiki，以指导如何处理生产问题。如果出现故障，待命人员无法迅速缓解问题，或者更糟糕的是，在他/她的值班期间，可能会增加更多问题。

# **3。沟通并做出决定**

一旦确定了技术债务，并且评估了相关的成本/风险，团队就需要决定他们是否真的想要将该债务添加到他们的系统中。更重要的是，他们需要决定如何记录债务，以及采取什么措施来偿还债务。

# **4。跟踪技术债务**

不管你计划如何记录你的技术债务，试着记录以下信息:

***描述:*** 什么，哪里，为什么(例如我没有写 wiki，因为我想把时间花在发布和测试用例上。)

***负责人:*** 做这项工作的成本(不做这项工作节省了多少时间，更新维基也许是一项 8 小时的工作)

***兴趣:*** 如果工作没有完成会发生什么，以及如果这导致以后的问题，额外的工作量。此外，记录这将在以后导致问题的可能性。

# **5。还债计划**

该功能发布成功，你很好地记录了债务，每个人看起来都很高兴。太好了！现在，你打算怎么还？它不应该作为一个低优先级的项目留在你的待办事项中。作为债务所有者，您需要积极跟踪行动项目，以确保债务得到及时解决。

# 结论

技术债务不一定是一件坏事，如果适当地添加到系统中，对企业和客户都是有益的。然而，挑战在于有效地评估利弊，更重要的是管理和跟踪利弊。随着时间的推移，在没有具体解决方案的情况下增加债务会增加企业的成本和风险。
