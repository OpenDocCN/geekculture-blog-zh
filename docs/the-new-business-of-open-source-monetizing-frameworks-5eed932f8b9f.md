毫无疑问，开源已经席卷了软件世界。尽管最初对一些人来说是一个问号，但是现在开源商业模式在很大程度上被认为是一个已经“弄清楚”的商业模式像 Elastic、Databricks、Confluent、MongoDB 和 Hashicorp 这样的公司已经建立了数十亿美元的销售开源软件的业务。这些组织的成功使得开放核心模型成为寻求向技术受众销售商业软件的公司事实上的标准上市模型。从很多方面来说，这并不奇怪——实际上，这是开发者免费增值。

这种模式对于上面提到的开源公司来说是非常成功的，但是在这一点上他们都已经很老了。下一波呢？

# 可操作性==货币化

这个问题的答案是微妙的。从根本上说，开放核心模型是基于一个简单的概念:保持软件对个人免费，但对任何与“操作化”相关的东西收费。这种方法的美妙之处在于，从功能上来说，“可操作化”或多或少直接翻译为“当一个人需要专业地使用产品时”这种货币化模式的第一次迭代是对基于角色的访问控制(RBAC)、高可用性、审计日志和支持等功能进行收费。这些特性是在任何规模的组织中使用这些产品的要求，因此也是可操作化的先决条件。第二次迭代是提供完全托管的服务。这是最纯粹意义上的可操作性:您实际上是在代表客户运行操作。

当这两种策略结合起来时，从初创企业到财富 100 强企业，这两种策略可以在整个市场范围内释放机会。这使得上述公司产生了巨额收入(Elastic 和 Mongo 都是 6.5 亿美元以上的 ARR 业务)。然而，值得注意的是，这一批公司有几个非常重要的共同点，使这一模式得以运行。也就是两件事:

1.  它们都是基础设施的一部分。我将基础设施定义为位于某个地方的盒子上，运行计算和/或存储数据的东西。
2.  它们都很难管理。他们需要“永远在线”构建和管理复杂=为运营付费的意愿很高。

对于满足上述两个约束的公司，有一个相当明确的剧本。所有这些产品的基本费用都与它们运行的底层资源(计算、内存、节点等)有关，另外还有安全性、高可用性等费用。该剧本源自多年来销售数据基础设施的方式，并针对开源和云时代做了一些小的调整。

不过还是那句话，下一波呢？

# 开源的新浪潮:框架

> 框架是他们的核心工作流程/生产力工具，而不是基础设施。它们不是坐在一个盒子上运行，而是提供脚手架，让工程师在开发和/或调试过程中更高效地工作。

最近一批开源公司的有趣之处在于，它们中的许多公司与以前的同类公司有所不同。尽管迄今为止几乎所有商业上成功的开源公司都将基础设施作为其核心，但今天“热门”公司中的许多公司都有些不同。他们仍然被高度技术化的受众使用，但是他们所围绕的底层开源项目是*而不是*一个基础设施。这是一个框架。框架是他们的核心工作流程/生产力工具。它们不是坐在一个盒子上运行，而是提供脚手架，让工程师在开发和/或调试过程中更高效地工作。他们的商业产品通常包含类似基础设施的组件(云作业等)。)，但都是从框架开始的。这是一个重要的区别，因为它对如何思考货币化有着巨大的影响。属于这一类的公司有盖茨比、Next.js/Vercel,·DBT/DBT 实验室、Pytorch Lightning/Grid.ai、air flow/天文学家、Dagster/Elementl 和远大前程/超导。

这一群体在采用方面处于中间阶段，但在货币化方面处于早期阶段。我认为，尝试使用传统的开放核心/托管服务方法来实现盈利将面临诸多挑战。

1.  一般来说，它们更容易独立运行，因为它们经常将任务转移到其他系统，而不是独立处理大量数据。
2.  他们通常不需要“永远在线”。你或*当然希望*他们永远在线，但他们不一定直接为顾客服务。

如果这家新公司无法从 OSS infra 公司获得盈利灵感，他们应该去哪里寻找呢？

我相信答案是 Github/Gitlab。简而言之，Github 是一个围绕代码协作框架的奇妙工作流体验。该产品的天才之处在于，它不仅使 git 更容易使用，还在相关但不相关的领域增加了大量的工作流价值。Git 只是构建更大东西的第一步——一个以极其有意义的方式增强它的协作平台。它从代码可发现性和审查开始，并进入各种各样的事情:问题跟踪、CI/CD、分支管理、代码分析、可审计性、代码扫描。这样的例子不胜枚举。

这个模型恰好也很好地符合“只有当人们操作软件时，才让他们付费。”你可以随心所欲地使用 git，而不需要把东西推送到 Github，我的人就是这样做的。但是当你需要和别人合作的时候？突然之间，你需要 Github(或者处理你自己的 git 服务器)。

如果说可伸缩性、安全性和高可用性是基础设施的可操作性，那么协作就是工作流的可操作性。归根结底，每个公司都是两者兼而有之——这不是二元的。成功的关键是了解你的产品倾向于哪一半。
