这些常见的数据孤岛正在推动安全数据湖的采用

原文：https://medium.com/geekculture/these-common-data-silos-are-driving-security-data-lake-adoption-1ee7fce7cfa?source=collection_archive---------29-----------------------#2021-06-28

网络安全从业者应该从分析行业中采用一个术语:“数据孤岛”。这是数据被留下的地方，没有被收集到真相的中心来源。筒仓使得挖掘数据以获得洞察力变得困难，打破它们是数据工程的一个重要部分。安全团队越来越多地面临云迁移和在家工作导致的数据孤岛，其速度超过了传统的安全信息事件管理(SIEM)。

为什么SIEM解决方案会造成数据孤岛？许多SIEM解决方案按每日接收量收费，这种模式与公共云基础架构生成的大量日志不符。虽然安全预算充其量是线性增长，但AWS和Azure记录的遥测数据比本地数据中心多得多。其他安全数据源也呈爆炸式增长，迫使安全团队接受其真实来源中的差距。在某些情况下，有价值的日志源被完全关闭，以避免超过SIEM供应商规定的每日接收限制。

许多SIEM产品导致孤立的另一种方式是通过多层体系结构。由于一方面要努力跟上数据量的增长，另一方面又要满足法规要求，传统解决方案会将数据复制到不同的归档层。恢复或“再水化”冷数据是一个可能需要几天或几周的过程。

即使没有规定每日收集配额或保留限制的SIEMs也可能会通过将安全数据与业务数据和企业分析团队隔离开来而造成孤岛。将安全性与企业数据堆栈的其余部分隔离开来，使得很难将有用的业务上下文应用于安全逻辑，并迫使cyber在没有公司的ETL和BI工具、流程和专家支持的情况下解决数据摄取和可视化挑战。

不应低估数据孤岛的有害影响。当安全数据源不集中时，它不能与其他数据相关联用于威胁检测。事件响应人员在调查可能的违规行为时，很难分别搜索不同的数据孤岛。当数据孤岛普遍存在时，很容易忽略可见性差距，直到它们被卷入事件中。

某些数据源经常被认为是安全数据湖计划的驱动因素。以下是高容量安全数据集，应集中起来进行有效的威胁检测和响应:

  • CrowdStrike原始事件数据:与过去的防病毒代理不同，端点检测和响应(EDR)代理几乎会跟踪端点上发生的每个操作。像CrowdStrike这样的解决方案就像一个“飞行记录器”,可以记录每个创建的文件、启动的流程和建立的连接。这些日志对于检测和响应威胁非常有价值——尤其是对于不在公司防火墙后工作的远程员工。拥挤的EDR通常是第一道防线，但是遥测数据的庞大数量阻止了它被收集到SIEM。相反，许多组织只收集警报记录(不到总数据的0.1%)，而将取证详细信息存储在CrowdStrike的环境中。CrowdStrike只保留丰富的传感器数据很短一段时间(7-90天，取决于你的计划)，事实上建议创建一个“丰富的传感器数据的离线副本，供
    在本地数据仓库或数据湖中使用，并与从其他系统收集的日志进行关联
    ”不将端点数据与其他安全和业务数据集结合使用，会极大地限制EDR对组织整体威胁检测工作的贡献，而不到一年的保留时间会危及违规调查。
  • AWS S3访问日志:近年来许多违规事件都涉及敏感文件从云存储桶中泄露。为了减轻这些事件的影响，了解哪些文件是由哪些用户从受影响的存储桶下载的至关重要。许多安全组织没有考虑到默认情况下该日志记录选项是禁用的。将S3访问日志存储在SIEM之外，或者一开始就不启用这种日志记录，是一种常见的数据孤岛情况，会妨碍云安全调查。
  • AWS VPC流量日志:与S3访问日志一样，VPC流量日志在AWS中默认是禁用的。这不是因为它们不值钱。除了威胁情报和异常检测之外，在系统不断出现和消失的云基础架构中发现攻击者还需要网络可见性。未能启用或收集流量日志是一个严重的差距，这通常是由SIEM摄取的高昂成本造成的。
  • Windows Powershell活动:勒索软件和其他攻击工具经常使用Windows Powershell脚本在Windows服务器上运行，并在加密受害者的文件时躲避检测。不幸的是，大多数IT组织不会跟踪Powershell活动，因为合法的Powershell事件数量太多。这使得安全团队无法确定使用了基于Powershell的恶意软件的漏洞的范围或“爆炸半径”。
  • Salesforce监控事件: Salesforce为安全可见性提供了一个实时事件流。这些数据有助于发现内部威胁，尤其是在与人力资源离职记录结合使用时，但通常被孤立在CRM中。眼不见心不烦。
  • 雪花访问历史: 访问历史是雪花中的一个新视图，可帮助客户生成精细的列级报告，以满足合规性审计。可以对其进行分析，以识别内部威胁或受损的用户帐户。如果公司的笔记本电脑遭到破坏，调查人员应该确认攻击者没有滥用所有者的雪花访问权限来下载敏感信息。
  • Sharepoint审计报告:微软Sharepoint支持关于文档访问的详细审计报告。违规调查应该能够轻松访问Sharepoint中的用户活动，并且在分发特洛伊木马文件的情况下，安全部门应该能够了解是谁下载了该文件。
  • Palo Alto GlobalProtect VPN活动:支持远程登录公司网络的基础设施越来越成为威胁参与者的目标。这使得VPN活动成为一个重要的分析数据源，不仅可以分析谁在进入网络，还可以分析他们在网络上做什么。将VPN数据存储在竖井中的一维方法很难发现被入侵的VPN用户。通过将VPN活动与端点遥测和HR记录结合起来，使用多维分析可以突出显示远程用户在无意或未经许可的情况下采取的行动。
  • 吉拉票证更新:虽然许多工程组织已经接受开发运维“左移”，但大多数安全团队并没有采取必要的措施来跟上。这意味着云基础架构中的频繁部署会转化为嘈杂的警报和安全分析师的倦怠。如果吉拉机票信息可以被纳入警报逻辑会怎么样？对Azure安全组的更改可能不需要敲响警钟，如果它在相应的吉拉票证中得到适当授权的话。相反，IAM策略中一个看似微小的变化，如果没有首先在票证中获得批准，就会引发安全调查。
  • 站得住脚的漏洞发现:服务器补丁级别与威胁检测和响应高度相关。在未打补丁的系统上检测到的任何情况都可能需要迅速上报，以控制事件的发生——即使该警报在其他情况下不会被认为是紧急的。尤其是在快速变化的云环境中，事件响应者应该知道系统是否短暂地处于未打补丁的状态。漏洞管理解决方案中孤立的漏洞数据阻碍了这些有价值的见解。

这远远不是通常孤立的安全数据集的完整列表。重要的一点是，以SIEM为数据平台的安全数据堆栈中的限制导致了危险的孤岛，这些孤岛变得越来越严重，影响越来越大。如果您在自己的环境中认识到了这些问题，那么是时候考虑安全数据湖架构了。
