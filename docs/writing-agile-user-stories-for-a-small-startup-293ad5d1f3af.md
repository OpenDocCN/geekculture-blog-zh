# 理解用户故事和敏捷开发。

用户故事是对任务或功能需求的描述。用户故事有时被称为卡片、任务、标签、故事、特性等等。用户故事通常用于敏捷软件开发团队。敏捷团队关注频繁和即时的可交付价值块。

如果你还不明白什么是敏捷团队，这可能有助于将他们与瀑布开发进行对比。例如，敏捷团队关注的是小而直接的交付，而瀑布团队关注的是前期规划。

尽管敏捷开发不仅仅是实践的集合，但是某些例程经常与敏捷团队相关联。而是一种不断迭代和改进的心态。这些惯例包括看板、每周冲刺、用户故事、站立等等。

要理解用户故事，你需要理解看板。看板类似于待办事项列表来组织任务。根据团队的需要，任务会经历不同的状态。常见的状态可能是 Backlog、In Progress、Ready for Review 和 Done。

# 如何写一个用户故事？

我对如何编写用户故事有自己的看法，我将与大家分享，但更重要的是，编写适合您团队的故事。不要仅仅因为其他开发人员或团队的建议对他们有用，就听从他们的建议。相反，找到对你和你的团队有用的东西。

软件行业对于编写有效传达特性需求的故事有很多看法。

您可以坚持严格的格式，比如:

*   作为一个`user role`我可以`action`这样`benefit`

或者，你可以写一些更侧重于技术开发的标题。

*   添加 CSS 预处理器。
*   修复损坏的 NPM 审计包。
*   加速缓慢的用户注释 API 调用。

有时你可能会有开放式目标的故事，尤其是研究任务。

*   探索 UI 框架的选项
*   探索搜索引擎优化选项

我经常看到人们说有一种正确的方法。我不这么认为。我认为所有这些都非常合适，取决于具体情况。

# 故事类型

在写作之前，我缺乏描述用户故事类型的术语。我发现了以用户为中心的故事、非用户故事和尖峰这些术语。术语只是术语，但这些符合我对故事类型的直觉理解，所以我将坚持使用它们。

## 关注用户的故事。

关注利益而不是开发者的实现。这些类型的故事通常适合这种格式:

*   作为一名`userrole`，我可以`action`这样做`benefit`。

## 非用户故事。

关注开发人员的实现或软件需求。这些故事倾向于适合较短的以开发人员为中心的格式。如果你以“作为一名开发人员”开始这个故事，你可能正在写一个非用户的故事。以我的经验来看，非用户故事通常适合于简明扼要的描述，而不是严格的格式。

如果你所在的团队强迫你坚持这种格式，你可以随意写:

> 作为一名开发人员，我可以坚持教条式的格式，这样我就可以让管理层高兴，并且已经构建了我那该死的特性。

然而，你也可能需要辞职，所以考虑忽略我的建议😆

## 尖刺。

秒杀重在探索知识。通常尖峰除了决定团队应该如何前进之外，没有明确的结果。这可能是研究如何遵守行业法规或研究技术解决方案的选项。

# 如何写出好的用户故事？

![](img/968f5e7b2fe8536c0decda47897b6614.png)

Warning! Dogma Ahead!

用户故事的存在是为了向处理期望任务的人传达信息。它们的存在不是为了遵循教条的格式或取代交流。你必须选择适合你的团队的格式，但是我确实有一些目标，我认为任何格式都应该努力实现。

## 一个故事不应该过度交流或交流不足。

组织类型倾向于更喜欢结构化的交流，并且经常想要给故事添加越来越多的细节。然而，以我的经验来看，形式化过程的愿望会导致额外的开销而没有任何好处。

然而，如果一个故事缺乏基本信息，那就是一个问题。例如，“改进风格”可能是一个没有帮助的故事，除非你的团队集体理解这意味着什么。

## 一个故事并不能代替交流。

无论你在你的用户故事中放入多少信息，你都不能取代交流。你不可能回答所有潜在的问题，你也不会通过尝试来节省时间。拥抱交流故事的需要。

用户故事永远不会取代你团队中的交流。如果你有问题，你需要谈谈。

## 故事应该是描述性的，而不是指令性的。

一个好的故事关注的是为什么而不是如何。他们关注的是目标，而不是手段。即使对于一个注重技术的故事来说也是如此。例如，如果目标是“减少页面加载时间”，但故事说的是“减少图像大小”，这就缩小了开发人员的关注点，并限制了他们创造性地使用解决方案的能力。

也就是说，在故事中提供推荐是非常好和有帮助的。只是要小心不要强迫开发者走上一条路。

## 一个故事应该写得很快。

在小团队中，故事应该写得很快。大型团队可能有不同的组织需求，但是小型团队应该是轻量级和快速的——这是他们的竞争优势。

如果你写故事的过程过于正式，考虑一下如何简化。

## 一张图胜过千言万语。

“修复错别字”是一个很好的用户故事，如果有一张带有错别字的图片的话。在你的故事中放入图片，传达的往往不仅仅是一个密集的需求列表。不是每个故事都需要图片，但如果一个故事适合图片，就添加它。

# 最后的想法

如果用户故事对你的团队有用，那么它就是好的。敏捷方法的目的不是采用特定的实践，而是关注对你有用的东西。因此，虽然我对一个好的用户故事的目标相当有信心，但是具体的方法和格式应该适合你的团队。

与 20 年前的软件公司相比，一个 24 小时的黑客项目会有不同的故事。所以没有一个正确的答案。

选择适合您团队的答案。
