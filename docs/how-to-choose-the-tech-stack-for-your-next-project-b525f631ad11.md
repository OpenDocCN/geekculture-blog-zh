随着每周都有新的 JavaScript 框架推出，Twitter 上的技术传播者不断推广新的首字母缩写词，为新项目选择技术堆栈已经成为新开发人员的压力来源。

对于有经验的开发人员来说，我在这篇文章中给出的建议可能是常识，但是对于更多进入技术领域的没有经验的开发人员来说，当时被出售几种不同的技术堆栈和框架自然会导致混乱和决策瘫痪，从而导致什么也学不到。这在 web 开发领域尤其普遍。新开发者被告知学习 React、Vue、Next.js、Remix、SolidJS、Svelte、Node 等的好处和痛点。新开发人员不应该关心 A 与 B 的利弊，他们只想建立一个 todo 应用程序或投资组合。

现实是，所有这些不同的编程语言、框架、服务等等。只是解决问题的工具。无论使用什么工具，软件工程都是关于问题解决的。成为一名软件工程师就是成为一名问题解决者，而不是一名特定的框架用户。当然，所有的工具和技术都是不同的。理想情况下，任何技术栈都旨在更有效地解决一组特定的问题。对于某些问题，有些栈比其他栈更好，所以软件工程师的工作是为工作挑选最好的工具**，并且当问题改变或有更好的工具可用时不要固执地改变工具**。

那么，如何为这项工作选择合适的工具呢？取决于工作，但一般来说，很少有通用的指导原则能让你达到 90%的目标。

让我明确这一点。这些天我们很幸运，像 React 这样的工具提供了很好的原语，允许开发人员有效地解决各种各样的问题，而不用担心。有些工具可能不是最好的，但大多数时候只要够好就够了。所以，大多数时候，选择一个工具实际上归结于对它的熟悉程度。如果你需要快速实现一些东西或解决一个问题，花一天时间阅读一个全新工具的文档可能是不值得的。

另一点:这是**而不是**关于 2022 年新开发者应该学什么的指南。同样，这是在选择工具来解决您的特定用例时要记住的概念列表。

# 尽你所能定义这个问题。

选择技术堆栈的第一步是定义您试图用它来实现什么。我相信这一点对于新开发人员来说并不是很好，因为他们试图解决的问题，比如构建一个简单的 CRUD 应用程序，可以通过几种不同的技术栈有效地解决。出于这个原因，我鼓励新开发人员通过解决更具体的问题和突破简单 CRUD 应用程序或 todo 应用程序的限制来逃离教程地狱。

在定义您试图解决的问题时，许多因素将推动您选择技术组合，例如:

*   如何以及在哪里部署
*   数据存储，如果需要
*   前端用户界面要求
*   等等…

认真考虑这些事情将有助于在你走错方向之前指导你的技术堆栈决策。即使这样，需求和问题范围也会发生变化，所以在这方面要随时准备灵活应对。选择能解决你定义的特定问题的技术，如果定义改变了，相应地调整你的技术栈。

# 开发者经验很重要。

开发经验差的技术栈和框架很糟糕，即使它们绝对是这项工作的最佳工具。如果你不喜欢开发体验，那还有什么意义呢？它只是把人们从软件开发中赶走，因为他们对糟糕的开发经验感到沮丧。假设您想要部署一个静态站点，并且您正在 AWS S3 和 [Vercel](https://vercel.com/) 之间做出选择。如果您还没有部署到 S3 的经验，Vercel 显然是更好的选择。选择学习 S3 是如何工作的，这给部署静态站点的最初目标增加了一个额外的问题。

好的开发者体验鼓励开发者做出好的决策。例如，在 TypeScript 与 JavaScript 的争论中，TypeScript 鼓励您使用类型，并在您的代码将导致意外错误时向您发出警告。普通的 JavaScript 没有提供类型系统，这将**总是**导致错误，没有帮助如何找到它。简而言之，具有良好开发经验的工具将使您成为更好的开发人员，这反过来将使您正在构建的任何东西变得更好，并且您在与工具的斗争中也不会那么沮丧。

# 选择错误的堆栈不是永久的。

如果你因为任何原因对框架或工具感到遗憾，重要的是要记住那些决定不是永久的，是可以修正的。我们大多数人都不会在软件失败会对用户造成伤害的情况下开发软件。重构和重写是软件开发的一部分，从长远来看，它提高了软件质量。

对于经历过冒名顶替综合症的新开发人员来说，感觉自己选择了错误的技术会伤害你作为开发人员的自信心。不幸的是，新框架和工具的营销材料经常试图通过让人们因为没有选择它而感到难过来获得用户。没有选择“正确的”技术组合并不是道德上的失败。

# 结论

那么，如何选择合适的技术组合呢？取决于工作和你感兴趣/熟悉的东西。我对初级开发人员的建议是，花更少的时间在不同技术之间做决定，观察每种技术的优缺点，然后开始构建。如果你发现你不喜欢你正在使用的东西，就换别的东西。

学习如何解决不同领域的各种问题自然会引导你学习各种技术。不要只是为了学习而在真空中学习一堆技术。
