# 1.可观察性[跟踪、监控和记录]至关重要！

您正在开发您的软件，并准备好部署它。所有的测试都通过了，测试覆盖率也达到了一个不错的水平。了解了这一点，我们可以部署我们的代码，并继续平静地工作。尽管这不是最佳方案(尽管很少)，我们的代码仍然可能失败。是啊！因此，开发人员需要一直观察他们的代码，并让它一直报告度量标准。如果出现任何故障，您需要让系统准备好将日志交付给您。正如安杰伊所写的:

可观察性至关重要。没有它，开发者是盲目的。它让我们有机会随时对系统中出现的每个问题做出反应。

# 2.使用“无服务器”和“有服务器”这两种方法是一个很好的实践。在这种情况下，我们从软件开发的两种方法中获益。

[无服务器](https://www.itmagination.com/blog/how-to-scale-node-js-apps-with-serverless)是一种运行应用程序的方式(看起来)不涉及任何服务器。当然，这是一个很大的简化——总是涉及到服务器；只有在这种情况下，您不需要对它们做任何事情，它们是预先配置好的。它被吹捧为新的黑色，除了…它不是治疗所有疾病的完美药物。首先，正如我们之前提到的，您不能配置底层服务器。你也不知道引擎盖下是什么。主要的缺点是这种方法的主要优点。您不需要配置任何东西，因此与其部署⇾担心，不如部署⇾忘记。

*无服务器解决方案和有服务器解决方案都有各自的优势。在现代系统中，通常将两种方法结合起来以获得大部分解。* **‍**

# 3.容器化一切！Kubernetes 是一项热门技术！

并不是所有的软件发展趋势都是好主意。你还记得 CoffeeScript 或者 Ruby 吗？很遗憾，我们有。幸运的是， [Kubernetes](https://www.itmagination.com/blog/360deg-tech-check-2-linkedin-security-breach-kubernetes-spending-github-copilot-facebook-bulletin-arm64ec) (K8S)看起来不像是要加入这两个人的悲伤之谷。k8 让 DevOps 专家的生活变得更加轻松。

通过将容器化和容器编排作为技术策略的核心租户，您可以获得以下好处:

*   您将能够轻松优化您的 IT 基础架构成本
*   由于无缝的可扩展性，您的用户可以期待更高的可用性和更好的 SLA
*   通过使用 Kubernetes，您的团队可以更轻松地使用云计算解决方案
*   因为容器和容器编排工具是技术无关的，所以您可以使用任何您想要的技术来构建一个更精简的团队
*   通过容器化，您不再遇到古老的“它在我的机器上工作”的问题
*   [Northwestern Mutual 案例研究| Kubernetes:](https://kubernetes.io/case-studies/northwestern-mutual/) 在收购一家金融科技初创公司后，Northwestern Mutual 不得不整合云本地和本地流程之间的工作流，更频繁地进行部署，并统一他们团队的运营，以确保他们的客户不断获得他们已经习惯的无缝体验。

    使用 Kubernetes 帮助他们将部署速度提高了 25 倍以上，从一年 24 次增加到 10 个月 500 次以上，计划内停机已成为过去，并且基础设施成本显著降低，这要归功于他们的 API 管理是部署在 Kubernetes 上的整体堆栈的一部分。

    *您可以在案例研究中了解更多关于这些要点的信息，包括向 AWS 的过渡、使用微服务、更新开发人员自主权，以及让他们的 450 万客户对其服务的质量和速度保持满意。*
*   [Capital One 案例研究| Kubernetes:](https://kubernetes.io/case-studies/capital-one/) 在 Capital One 的案例中，底线是一个很大的话题。他们的估计表明，如果不使用 K8s 的自动和轻松扩展的能力，他们的 AWS 基础设施成本将很容易增加两倍甚至四倍。

    他们通过使用 Kubernetes 看到的其他好处是新产品的上市时间，现在只需 2 周，而以前可能需要 3 个月或更长时间。

    开始看 Kubernetes 对于他们发展的主要原因？Capital One 的团队希望提高处理流数据的速度，这些数据用于欺诈检测和信贷决策领域的关键决策，以及对银行日常运营至关重要的其他大数据和机器学习应用。

    *关于这几点，部署速度提升，K8s 如何帮助统一 Capital One 的开发环境，更多在案例研究中。*

除了显而易见的好处之外，我们将留给您一些案例研究，我们希望这些案例研究能够启发您，帮助您决定使用 Kubernetes 是否适合您，并展示更深入的好处:

*   [Prowise 案例研究| Kubernetes:](https://kubernetes.io/case-studies/prowise/) 在包含许多产品的复杂开发环境中，应用程序版本之间的停机时间为零，新部署只需几小时到几秒钟，新版本的发布速度提高了 3 倍。

# 4.说到软件架构，我们应该分而治之

大型的独石有些像是昨日黄花。它们困扰了开发人员很长时间，但现在不是了。将庞大的单一代码库分割成较小规模的应用程序是一种新的做事方式。它使你的应用程序防火，减少出错的频率，并在出错时使应用程序更加安全。不利的一面是，应用程序变得更难测试，而且测试需要更多的资源。对于较小规模的团队来说，保持一个整体仍然更有意义。

*将单片应用划分为独立的微服务。*

# 5.开源和自由软件是未来的发展方向。

[React](https://www.itmagination.com/blog/360deg-it-check-41-react-18-kentucky-bitcoin-astro-chrome-100-vs-code) 、 [Angular](https://www.itmagination.com/blog/360-it-check-47-next-js-angular-jquery-microsoft-build-duckduckgo) 和 Zuul，分别来自 Meta(曾经是脸书)、Google 和网飞，是无数开发者每天在工作中使用的工具。如果没有这些组织免费向所有愿意使用它们的人发布的工具，每个人的工作都会变得更加困难。无数的服务将不会出现，因为编写这些应用程序太难或太耗时。这都是因为，在编写它们之前，人们必须弄清楚如何独自编写用于扩展的前端，而不分享学到的经验将是非常低效的。

这就是为什么我们必须赞扬开源&自由软件的维护者、创造者以及所有其他为这类软件的创建和维护做出贡献的人。

*创造一种工具/技术并使其开源(或免费)会给组织带来永恒的荣耀。*

# 6.使用架构模式。

软件开发有一个共同的规则——不要多此一举。知道我们可能已经像其他人一样面临同样的问题，这条规则就变得更有价值了。这就是为什么来自世界各地的工程师和开发人员使用架构模式来构建他们的项目——而不是浪费时间去思考如何找出别人已经提出的解决方案。

许多现代软件都使用像 CQRS 和事件源这样的模式。不要重新发明轮子，使用模式。

# 7.编程语言在不断发展。

事实上，我们有越来越新的编程语言并不奇怪。都是来了又走，走了又被别人代替。没人再用 Algol 或 Pascal 编程了。然而，一个老前辈 C 仍然存在，尽管这是一个值得单独探讨的话题。

一个值得注意的方面是这些年来它们的进化方式。起初，命令式语言是仅有的语言。然后，面向对象语言的热潮来了，现在，有些人可能会说，它们正在被更加灵活的语言所淘汰，这些语言混合了一些命令式的、函数式的和面向对象的特性。

语言进化的方式越来越独立于我们工作的系统。现代语言是跨平台的。由于 DevOps 的成长，语言选择变得越来越不重要。

# 8.由于现代基础设施，复杂性正从应用程序转移到外部平台。

云提供商和相关技术取代了地下室物理服务器上的传统基础设施。我们有虚拟机即服务、数据库即服务和许多其他基础架构元素即服务。软件解决方案中的主要规划已经转移到基础设施的高层次设计，因为许多都可以基于它实现自动化。此外，我们还有容器和容器编排。它接管了复杂性，因为我们可以将系统分成更小更简单的部分。

应用程序代码变得更加独立于平台。然而，复杂性在于基础设施和操作。应用程序开发人员越来越关注业务逻辑。DevOps 工程师处理其余的。

# 9.SCRUM！=敏捷

采用特定的过程通常会导致习得行为，最终会形成习惯。至少，理论上是这样。

然而，在某些情况下，过程停留在人们跋涉的过程中，只是为了走过场，但行为从未发展。这样想，你见过多少开发团队经历了所有的 Scrum 仪式，但实际上没有以敏捷的方式工作？太多了？我们同意。

那么你能做什么呢？首先，团队买入，这永远是需要建立的第一步。如果你的团队看不到使用这种方法工作的价值，那么从长远来看，所有的过程和惯例都不会有太大的改变。

第二步是确保你有一个优秀的 scrum 大师和项目经理，以确保好的实践被向前传递，任何反对意见被

第三步是要意识到:在没有任何价值的情况下，强迫人们接受敏捷价值观 Scrum 框架，会很快让你一事无成。我们已经在标题为[“Scrum 并不是所有 IT 项目的答案(IT magination . com)”](https://www.itmagination.com/blog/scrum-isnt-the-answer-for-every-it-project)的文章中详细讨论了这一点以及更多细节。

*SCRUM 可以是敏捷的，但它不能保证敏捷。敏捷来自于行为，而不仅仅是过程。*

# 10.持续安全性

正如我们以前多次提到的，安全不能是事后的想法。我们不能简单地“留待以后”检查应用程序的安全问题必须集成到开发运维流程中，并且从第一天起就集成到开发流程本身中。幸运的是，我们几乎没有什么工具可以让这个过程没有摩擦。其中一个就是 [Snyk](https://snyk.io/) 。这是一个全面的工具，可以“发现并自动修复代码、开源依赖项、容器和基础设施中的漏洞，就像代码[……]”

我们必须在开发周期中应用安全检查程序。安全是信任的基础，信任是未来的货币。

# 11.审计云提供商的服务价格

因为三个主要的云提供商几乎没有竞争，他们提供的服务的差异(或多或少)是任意的。实际上，我们可能看到的唯一区别是服务价格的不同。这就是为什么对这一特定供应商有偏见不一定是坏事。实际上，大多数情况下，几乎没有区别。

选择你觉得舒服的，并且已经知道的供应商。边走边评估，不要害怕改变。

*云提供商没有虚拟竞争，没有成本套利。云基础设施成本非常依赖于通货膨胀和经济衰退。*

# 12.一切都可以“作为服务”来完成。

平台即服务、基础设施即服务、数据库即服务、软件即服务、后端即服务……我们为您提供了更多示例，您应该能够理解我们的观点。你能想到的都可以让第三方来做，卖给你。

使用这些服务是一种权衡。您放弃了一些控制，以变得更精简，能够更快地迭代，以及预先节省一些钱。

*由于云提供商和无服务器方法重要性的增长，每一个软件都可以作为服务来完成。*

# 13.每个人都使用 Visual Studio 代码。

Visual Studio 代码风靡全球。有微软的支持，有开放源码许可，用 TypeScript 编写，并允许功能的简单扩展，这些都是伟大的决定。到目前为止，文本编辑器是现代程序员中最受欢迎的选择。其他选择，如基于 Intellij 的集成开发编辑器(IDEs)或 Vim，都在 Code 的阴影下，尽管 [JetBrains 的舰队](https://www.itmagination.com/blog/360deg-it-check-41-react-18-kentucky-bitcoin-astro-chrome-100-vs-code)可能会改变这一点。

得益于多种扩展和定制工具，VS Code 成为开发人员中最受欢迎的 IDE。

# 14.TensorFlow 目前被广泛使用。

TensorFlow，谷歌的机器学习框架是程序员中非常受欢迎的选择。首先，它是 GitHub 排名前 20 的最受欢迎的存储库。然后，有多个端口，[包括 JavaScript one](https://www.itmagination.com/blog/javascript-frameworks-part-3-data-science) ，团队在他们的 [React Native](https://www.itmagination.com/blog/cross-platform-mobile-apps) 应用中使用，或者在 React 或任何其他 [JS 框架](https://www.itmagination.com/blog/javascript-frameworks-part-1)中使用 [web 应用。这提供了巨大的灵活性，并允许团队将解决方案嵌入到许多解决方案中。](https://www.itmagination.com/blog/react-18-what-changes-does-it-bring)

*借助 TensorFlow，我们可以在 web 应用中实施人工智能解决方案。培训模型由图书馆提供。开发者应该专注于训练他们。*

# 15.一个很好的长期招聘策略是雇佣初级员工并对他们进行培训。

雇佣初级员工是一个很好的长期策略。虽然没有适合所有公司的“最佳策略”，但雇佣并培训初级员工无疑是培养和留住内部人才的最佳方式之一。

雇佣初级员工是一种随着时间慢慢扩大团队规模的好方法，而且与雇佣那些可能已经习惯了他们的方式的人相比，建立一种更容易塑造的内部文化。大三学生也能提供新的视角，更了解当前的趋势。

有些情况下这并不理想，例如当你的公司需要快速扩展和开发新功能时。如果您有一个小的内部团队，由于不切实际的开发期望，他们总是试图赶上他们的积压工作，这也不是最好的。在这种情况下，[雇用外部技术合作伙伴来帮助开发，同时扩大内部团队可能是一个很好的折中解决方案。](https://www.itmagination.com/blog/outsourcing-versus-in-house-software-development-pros-cons)

作为一种策略，雇佣初级员工来训练他们并非没有陷阱。你团队中的初级员工没有被之前的公司审查过，他们没有工作经历，这很可能是一次成功或失败。不幸的现实是，尽管这种策略在适当的薪酬方案下可能非常有效，但初级员工可能会发现，他们只需跳槽，就可以获得两倍、三倍甚至四倍的薪水，而不是等待或争取升职或加薪。

这就是为什么拥有透明的薪水和薪级表如此重要，以向人们展示他们可以在职业道路上前进的地方和方式。这就是为什么拥有优秀的入职培训计划也非常重要，以确保花在培训新人上的时间得到合理利用，并使导师和学员都受益。

根据大量的研究，软件开发人员长期招聘的一个很好的做法是雇佣没有经验的工程师，并把他们训练成组织的方式。
