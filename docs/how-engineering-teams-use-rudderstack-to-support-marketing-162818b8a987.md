这里概述了工程团队使用 RudderStack 从数据层支持营销的具体方式。

# 步骤 1:整合和标准化第一方数据收集

当涉及到第一方数据时，营销面临的一个核心挑战是事实的不同版本。在比较两种不同的营销/销售工具时，症状往往会显现出来。你可能听说过“Google Analytics 中的数据与 HubSpot 中的数据不符”，或者被问到“为什么 Klaviyo、Salesforce 和我们的 Looker 报告之间有这么多差异？”

数据差异几乎总是会发生，因为各种工具使用不同的技术机制来收集数据，并且每个工具的定义和格式都略有不同。回到上面的例子，Google Analytics 和 HubSpot 都需要你运行它们的 JavaScript 代码片段，它们收集浏览行为的方式也不同。Klaviyo 和 Salesforce 具有原生同步功能，但自定义字段、列表成员资格和其他数据点不能很好地转换。Looker 数据来自仓库，其中的数据可能已被分析团队清理干净。随着堆栈中工具数量的持续增长，这个问题变得更加严重。

**解决方案是将追踪整合到一个单一来源，并标准化数据收集。**

这个问题的本质要求在源头进行标准化和简化，这正是 RudderStack 的 SDK 所支持的。有了 RudderStack，您只需跟踪一次，我们简单的 JSON 有效负载遵循标准化的模式。

这里有一个例子。假设你是一家电子商务公司，营销团队使用优惠券让人们注册他们的每周交易简讯。营销部门希望在新闻通讯注册活动中添加额外的数据。具体来说，他们想知道促使注册的优惠券的名称。通常，这将是一个巨大的痛苦，需要为每个跟踪注册的目的地(Google Analytics，Klaviyo，Salesforce 等)编写稍微不同的代码。)…这还没有考虑更新广告平台的转化率(我们将在下面讨论)。

在 RudderStack 中，您只需检测一个注册事件并识别新用户。您可以发送名为“时事通讯 _ 注册”的. track 呼叫。要添加优惠券代码，您只需获取值并将其添加到 track call 的属性中，这将向每个营销工具发送相同的标准化有效负载(下面将详细介绍集成)。您还可以识别用户，以便在营销、电子邮件和 CRM 工具中创建联系人记录。

![](img/8d4a53728fdf9e1874d9b93099ef5c08.png)

对于开发人员来说，这意味着大大简化了第一方跟踪工具的工作流程，更重要的是，简化了正在收集的数据的标准化。这也意味着控制——在谷歌标签管理器中不再有惊喜标签……事实上，你可以完全取消谷歌标签管理器(哈利路亚！).最重要的是，模式可以在您的代码库中进行版本控制，因此您可以确切地知道正在收集什么数据。

对于您的营销团队来说，这意味着跨工具的数据一致性，以及当他们向您和工程团队请求数据时更快的周转时间。

# 步骤 2:从等式中移除积分工作

即使您将数据收集标准化，您仍然需要将它发送给市场营销使用的所有工具。通常这意味着开发人员必须做大量的定制 API 工作来发送标准化的有效载荷。

RudderStack 消除了等式中繁琐的积分工作。您仪器的标准化事件可以自动发送到您堆栈中的任何目的地，包括营销工具。没错——不需要集成工作。让我们回到上面的“时事通讯 _ 注册”和用户识别示例。

![](img/f094c8e115e3a2cc59e10758889eb100.png)

当 identify 调用运行时，RudderStack 将在所有适用的下游工具中创建用户记录。在这种情况下，用户将在 Klaviyo、Salesforce 和您在 RudderStack 中作为目的地添加的任何其他下游工具中创建。如果您将您的数据仓库添加为目的地，我们还将更新用户表，以便您的 Looker 报告具有最新的数据。

“时事通讯 _ 注册”。跟踪呼叫是一个行为事件，并不适用于每个目的地。在这种情况下，营销部门希望将事件发送到 Klaviyo(以便他们可以将这些注册划分到一个列表中)和 Google Analytics(以便他们可以分析转换漏斗)。同样，RudderStack 将自动用事件信息填充 Kalviyo 和 Google Analytics。

# 步骤 3:简化 ad 平台的客户端和服务器端转换管理

对于开发者来说，唯一比与营销的云工具进行定制集成更糟糕的事情是处理将转换发送到付费广告平台的巨大混乱。

在脸书、谷歌、Pinterest、必应等网站上进行专门的转化活动。需要持续关注营销运行实验，这意味着大量定制代码或痛苦的故障排除和谷歌标签管理器中复杂的排序。呸。

在 RudderStack 中，您的*现有事件*可以作为转换直接发送到广告平台，无论是客户端还是服务器端。所以，让我们说，营销想要创造一个再销售受众，为每个订阅时事通讯的人提供广告。要做到这一点，您只需将您的广告平台添加为目的地，然后将您现有的“简讯 _ 注册”活动映射到平台中相应的转化(例如，对于脸书，您可以将“简讯 _ 注册”映射到潜在转化)。

# 步骤 4:使用转换动态定制和修复目标集成

无论你如何简化你的数据流向市场营销工具，市场营销的需求会随着他们在实验中的学习而改变。这意味着不可避免地，数据有效载荷将需要更新，并且需求将改变，即使是在逐个目的地的基础上。

例如，假设营销团队希望将``most_recent_coupon`'值推送到 Salesforce，但他们创建的自定义字段是“`Latest_Coupon_Code__c`”。您可以在 RudderStack 转换中使用简单的 JavaScript 为 Salesforce 目标定制有效负载，而不必向原始事件添加另一个键。在这种情况下，您可以将“`most_recent_coupon`键重命名为“`Latest_Coupon_Code__c`”。

因为转换是用 JavaScript 编写的，所以没有限制，我们的客户已经找到了数百种用途，从定制到即时修复。

```
export function transformEvent(event) {
if (event.event == 'most_recent_coupon') {
event.event = 'Latest_Coupon_Code__c';
}
if (!event.properties.sale_item) {
event.properties.sale_item = false
}
return event;
}
```

一旦你完成了底层的数据管道工作，你就可以开始在有趣的事情上与营销部门合作了——这些项目真的可以为公司带来进步。这里有几个例子。

# 将丰富的数据从仓库直接发送到营销工具

使用 dbt 之类的工具在仓库中构建丰富的客户视图变得越来越常见，但是将这些丰富的数据点放到云工具中是一件非常痛苦的事情。使用 RudderStack 的仓库操作特性，您可以将表转换为有效负载，并填充相同的下游工具。

例如，您的仓库中可能有一个客户列表，他们购买了您公司拥有的另一个品牌的产品，而营销部门希望将他们排除在 Klaviyo 的某些电子邮件活动之外。通过仓库行动，你可以在你的仓库中创建群组，并将其推送到 Klaviyo，这使得营销很容易建立正确的逻辑。

# 直接发送受众和群组以添加平台

数据工程团队通常可以访问构建最有趣、最有价值的营销受众所需的数据。比方说，你已经用第三方意向数据丰富了高生命周期价值的客户，并在你的仓库中拥有了这一群体。

RudderStack 可以将受众直接推向广告平台，如脸书客户受众。
