## RESTful API 设计简介。

![](img/1d296ae1537420a0651459f266ee10a3.png)

那么什么使得 API 是 RESTful 的呢？我相信 REST API 或 RESTful API 这个术语在您的编码之旅中已经无数次出现在您的屏幕上了。本文介绍了 RESTful API 设计原则。

在深入研究这六个约束之前，我将简要描述什么是 API 以及 REST APIs 是如何工作的。API 代表应用程序编程接口。REST 代表 **RE** 表象 **S** 状态 **T** 转移。它是一个 API 体系结构，具有客户端-服务器通信和功能的设计原则。

RESTful APIs 通过 HTTP 请求进行通信，以在资源中执行 CRUD(创建、读取、更新和删除)操作。这些操作对应于方法请求(GET、POST、PUT、DELETE)。POST 请求创建记录，GET 请求读取记录，PUT 请求更新记录，DELETE 请求删除记录。遵循以下六个约束的 API 被认为是 RESTful 的。

# 1.客户端-服务器独立性

在 RESTful 设计中，客户机和服务器必须彼此不同。客户端使用统一资源标识符(URI)请求信息，服务器通过 HTTP/HTTPS 响应请求。客户端和服务器都应该独立发展。这种独立性允许两端的可伸缩性。

# 2.无国籍的

无状态约束意味着客户端状态不会存储在请求之间。来自客户端的请求为服务器提供了足够的信息来理解请求，而无需记住任何客户端状态。不存储关于客户机请求的数据可以提高可伸缩性。

# 3.统一界面

统一的接口确保了跨平台的标准客户端-服务器通信。使用 REST API 架构，对同一资源的请求应该具有相同的数据。以及来自服务器的响应状态代码，例如 200 ok、401 unauthorized 等。

# 4.分层系统

客户端并不总是与服务器直接通信。交流可能有层次。对于具有几层的架构，每层只知道它的直接通信层。分层系统允许完成复杂的任务，而不需要理解接收响应的复杂性。

# 5.可缓冲的

可缓存数据是响应中的信息，可以存储并在以后使用。

服务器响应应该声明数据是否可缓存。为了提高网络效率，来自服务器的响应中的数据被隐式或显式地标记为可缓存或不可缓存。对于可缓存的数据，客户端可以为将来的请求重用响应。

这种约束提高了客户端的性能，同时减少了负载并提高了服务器的可伸缩性。
