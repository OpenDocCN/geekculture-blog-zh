你好。我想说实话:我真的对 React 和它的生态系统感到厌烦。一天中你会看到多少次“未捕获类型错误:无法读取未定义的属性”的错误？你有没有想过有更好的方法来编写在生产中不会崩溃的代码？这是我探索了近一年的话题，以下是我的探索。

# 拉姆达

对我来说是从拉姆达开始的。这是一个不错的 JavaScript 库，有很多有用的函数。这允许，有时甚至强迫你编写可组合的代码。这是指开发独立的代码片段，然后使用“组合”功能组合它们，以产生有价值的结果。

我仍然每天都在使用这个库。如果你想开始编写可组合软件，我推荐你使用它。当然，这不是一个理想的解决方案，而且还有一些问题，比如理解复杂的代码组成，有时甚至是性能。

所以，我没有解决核心问题——“无法读取未定义的属性”。这是因为我们需要一个好的解决方案来进行代码静态分析。在这里，我决定进行下一步…

# 以打字打的文件

是的，这是炒作。围绕它有大量的谈话和文章。而且写更安全甚至自描述的代码真的是一件很好的事情。这看起来比使用纯 JavaScript 的 Ramda 好得多:类型注释、接口，甚至静态代码分析！

这就是我想要的——在编译时检查代码，并在投入生产之前发现可能的问题。它很好地解决了这个问题:您可以找到类型不匹配。但是为了防止运行时崩溃，您需要深入了解并遵循良好的实践:

*   正确处理对象修改
*   遵循总体性原则
*   使用“也许”和“要么”单子

第一点看起来很简单——在修改对象之前进行浅层复制。但是另外两个呢？

**总体性原则。**简而言之，这是你保证你的函数不会抛出错误，并且总是返回一个预期类型的值。[让我带你到这里了解更多细节。](https://kowainik.github.io/posts/totality)

**也许&任一单子**。这些单子是很强大的东西。它们允许您用包装器分别包装空值和错误，以防止运行时崩溃并从意外情况中恢复您的应用程序。更准确地说，错误是安全应用程序中的预期情况:计划突出的情况是一件好事。“Monad”和“要么”包装器允许您进行这种规划。[点击此处了解更多详情。](https://mostly-adequate.gitbook.io/mostly-adequate-guide/ch08)

而且，如您所见，我仍然对 TypeScript 提供的类型安全不满意。所以我继续寻找更好的解决方案，但不是那些大肆宣传的、行业强加给我的解决方案。

# 哈斯克尔

老实说，我对 Haskell 感兴趣只是为了理解一些经典例子上的函数式编程。我没有用 Haskell 编写产品代码的计划。而且我可以推荐给你学习 Haskell。尝试不同的东西是件好事。Haskell 与我过去几年在日常编程中所做的相比有很大的不同。

我还在了解哈斯克尔的路上。我认为有很多有用的概念我们可以带到 JavaScript 世界。

# 榆树

我听说过很多。我想尝试一下。我甚至开始阅读官方文件几次，然后放弃，因为我认为这是浪费时间。但是今天我又开始读 docs 了，现在是认真的打算。在经历了一些 Haskell 之后，我看到了 Elm 这样的语言的强大之处:

*   易于学习并开始构建应用程序
*   实现基本的 FP 概念:currying、Hindley-Milner 符号、复合等等
*   支持静态代码分析，并提供保持类型安全的工具

我认为这个列表可以扩展很多。但是在我已经走过的路之后，还有一些对我来说最明显的东西。这看起来是一项很好的技术，将帮助我构建可靠的应用程序，在运行时不会崩溃。我看到在我面前有一条新的漫长的路，我要学习新的概念和东西。我相信埃尔姆并不是这方面的最后一个观点。

# 摘要

当然，我不会完全放弃 React。React 是一个强大的库，可以快速构建漂亮的 ui。我喜欢一个庞大的社区。当然，还有很多有趣的项目。但对我来说，这并不意味着这是一个结束。这只是一个开始，因为构建 ui 的方法正在发展，看起来它正在慢慢走向函数式编程…所以也许 Elm、Svelte 等技术是前端世界的未来。

祝你们好运，伙计们！
