我做过很长时间的后端，后来在职业生涯中转行做前端。如果有必要，我的技能让我可以接受任何一方的任务，但不是同样的快乐。有人告诉我，“不管喜欢与否，你都是一个全栈开发人员”，我不同意这种说法，这促使我就这个主题多写一点。我还注意到全栈开发人员的职位有所增加，随后我和周围的人进行了几次交谈，包括在 LinkedIn 上的一次长时间交流。这是一个有趣的话题。

# TL；速度三角形定位法(dead reckoning)

底线是——如果你什么都擅长，那你什么都不擅长。让软件工程师了解他们工作的不同方面是很好的，这是必须的，但是要求他们把时间集中在所有事情上不是他们喜欢的事情，也不会给公司带来好处。

## 为什么它不起作用？

*   你不能指望人们在每件事上都很优秀。不可能。
*   99%的开发人员偏好前端或后端。
*   这两个领域都很复杂，他们需要时间来做好。
*   终身满栈者很少对任何话题有很深很深的了解。
*   全栈团队不会增加灵活性或公司的敏捷性。
*   质量经常受到影响。
*   在大多数情况下，开发人员会更喜欢半栈，但是他们经常被迫做所有的事情。让人们做他们喜欢的事情。
*   我了解到，大多数自称全栈的人心里都有后端。

全栈在 15 年前是可能的，那时你只需要知道语言本身。如今，堆栈如此复杂，你已经可以看到每个领域内的进一步专业化。在您需要帮助的地方寻找优秀的专家，结果，您会发现一些人在必要时可以使用全栈。不要强迫人们做他们不想做的事情，确保你的团队关心结果，即使结果超出了他们的舒适区。记住，你不能通过解决症状和重复“我们必须敏捷”来掩盖组织中的根本问题。

![](img/bc8298e124f9f34cdd3f49974c162dd8.png)

这篇文章的目的不是贬低全栈开发者，相反，我想强调这个术语是多么模糊和不准确。这些天我们的开发堆栈变得非常复杂，在招聘广告中声明“前端开发人员”已经太笼统，不够描述性。说“满员”会让事情变得更糟，减少公司找到他们要找的人的机会。

让我们试着关注全栈意味着什么，以及公司如何证明他们对如此广泛的技能集的搜索是合理的。我将分享一些我在不同工作场所的经历，并以一些公司如何改变以使事情对他们自己和他们的工程团队更好的建议作为结束。

# 神话中的全栈开发者是谁？

一个在前端和后端开发方面具有天赋和同等技能的人。能够从数据库开始，通过后端逻辑、API、前端、布局到 CSS 和动画，交付应用程序的垂直切片。她对某个特定领域没有强烈的偏好，可以在这两个领域做出关于架构、设计、工具和技术产品方向的短期和长期决策。

# 这样的人存在吗？

有些人相信外星人，有些人相信雪人，我相信世界上有少数人认为值得信赖的政治家是存在的。我正在寻找一个被称为“全栈开发人员”的神秘人物，他将在 web 开发的所有领域(或任何其他开发领域)拥有令我惊叹的知识。

在此基础上，我对这些问题的回答是“不，不是真的”。

我还没有见过一个人能够同等地交付前端和后端代码。有些人说他们乐于在任何一方工作，但在与他们交谈后，我很快发现他们更喜欢在一个领域工作而不是在另一个领域。给他们后者，他们可能会花更长的时间来交付，质量也会受到影响(很可能他们不会承认)。每个人在某个特定领域都更强，公司应该努力首先关注人们的优势。

> “[……]全栈 dev 更偏向于一方，他只是在玩弄另一方。所以其他人需要在他玩耍的一边做一些清洁工作。”

后端技能不仅仅是写 PHP 或者。NET 代码。前端比 JS 和 CSS 多一点。如果一个公司让一个人负责所有的事情，他们就失去了在某个特定领域更深入的知识和经验的宝贵投入。

# 那么，为什么公司要找水管工-理发师-飞行员呢？

公司决定寻找这样的人加入他们的软件工程团队有不同的原因。较小的组织往往倾向于雇用全栈开发人员，因为他们需要他们的产品来快速赚钱。他们(还)雇不起很多人只做“一种”工作。证明应用程序有效的证据通常是“足够好”来发布它并继续前进。在这个阶段，质量很少是优先考虑的问题。

有时，较大的工程团队(通常在较大的公司中)倾向于相信雇佣全栈将解决他们所有的问题。当人们可以做任何事情时，他们应该更加灵活。令人惊讶的是，他们经常重复，“我们需要敏捷！”证明全筹码的合理性。恐怕我不得不反对迄今为止我听到的大多数借口，值得更详细地讨论它们。

**省钱** —雇用全栈人员省钱(显然)，因为他们可以做任何事情。也许这是真的，但同时，你不应该期望这两个领域的代码质量处于同一水平，更不用说他们的时间是有限的。这可能是一个不错的短期策略，但长期策略呢？

**一个人完成一个任务/功能更快(垂直切片)**——显然，他们什么都不用等，也没人阻拦。有一种误解认为，在团队中工作意味着人们总是在开始任务之前等待其他人完成。我有一种感觉，持有这种观点的人从来没有在一个高效和相互支持的团队中工作过。顺便说一句，大概十年前我在这样一个团队工作的时候，我也会认同这一点。

**他们更了解整个平台，他们可以快速“加入”这些碎片**——的确，他们可以，但更大的概率是只有他们能做到。它不可扩展，也不鼓励共享或编写适当的文档。在这一点上，没有人知道事情是如何运作的，如果我们无所不知的大师去度假或被流星击中会发生什么？

**如果有人离开，他们可以填补—** 是的，这是真的，希望之前的那个人留下了一些文档，并分享了她正在做的事情。全栈可以在短期内更容易地取代前端或后端特定的角色，但这不会持续很久。像这样的情况很少见，公司要做好准备。另一方面，任何高级或首席开发人员应该暂时在另一个领域提供帮助。

> “一个团队中的 10 个全栈开发人员听起来比 10 个半栈开发人员更好，没有人想成为坐冷板凳的半栈开发人员。”

**在 sprint 团队里更容易给他们分配故事，他们总会有事做——**远离真相。是的，分配任务更容易，但它变得混乱，从长远来看永远不会有回报。如果一个团队最终没有足够的工作，团队成员无所事事，那么公司在文化、信任和组织方面存在根本问题，而不是开发技能。

# 那么为什么公司要把精力放在半栈开发者身上呢？

公司不应该寻找符合所有条件的工程师，但通常情况下，他们就是这么做的。相反，他们应该把重点放在那些喜欢特定技术组合的人身上，拥有其他语言/领域的经验是额外的奖励。无论如何，他们应该期望从他们雇佣的任何高级开发人员那里得到它。前端家伙必须有一些关于后端和什么是可能实现的知识。很有可能他们在过去构建了一些 API 并使用了数据库。世界不会在他们的鼻尖结束，他们需要了解他们做的事情如何适应更大的图景。顺便说一下，这是双向的——后端人员也应该对前端有所了解。

> “当一个项目达到一定规模时，鉴于我们理解和应用最佳实践的能力有限，维护这种专业知识(全栈)会受到回报递减的影响。”

团队应该学会如何信任和依赖彼此。团队成员一次专注于一个领域，但他们对另一个领域有所了解并有一定的经验。他们不需要深入了解垂直特征切片的每个细节。传递故事和一起工作是整个团队的责任。

**他们应该懂语言，而不是图书馆。**当你开始遇到意想不到的问题，需要扎实的基础语言知识时，他们的专业知识将变得至关重要。团队应该总是为工作选择正确的工具，而不是试图围绕他们知道的唯一工具来适应工作。当团队需要快速行动和学习新事物时，了解语言(不仅仅是框架)将是至关重要的。

他们可以更快、更好地交付产品，同时兼顾性能、设计、未来技术方向和架构。产品团队没有时间(或理解)这些东西，所以它们需要融入工程师的日常工作中。

他们是团队的一部分，可以与另一边同样博学的队友进行讨论，学习并提出独特的想法，而不受他们在所有平台上的延伸的限制。拥抱技术专长，鼓励交流，建立团队，共同工作，分享思维和行为的多样性。

如果在一个团队中有太多人被分配做类似的工作，他们就会开始竞争，这可能会成为一种不健康的竞争。当你看到人们为自己囤积信息，试图让自己变得不可替代时，这是事情没有朝着正确方向发展的迹象。

# 对此能做些什么呢？

当你有健康问题需要手术时，你不会去找你的全科医生做手术，即使他声称他能做。你想要一个专家，一个专家，一个外科医生。你需要一个了解人体并且有丰富经验的人来切开、固定和缝合人体。你冒着生命危险，你想得到好的照顾。这并不意味着 GP 不能做这项工作。外科医生更有可能代替全科医生来完成复杂的手术。我希望你明白我在做什么比较……虽然编写 web 应用程序不是一个危及生命的操作，但你希望它做得正确，并相信“外科医生”。在他们当中，你总会在需要的时候找到偶尔的“可以成为全科医生的外科医生”。

全栈在 15 年前是可能的，那时你只需要知道语言本身。如今，堆栈如此复杂，你已经可以看到每个领域内的进一步专业化。在您需要帮助的地方寻找优秀的专家，结果，您会发现一些人在必要时可以使用全栈。不要强迫人们做他们不想做的事情，确保你的团队关心结果，即使结果超出了他们的舒适区。记住，你不能通过解决症状和重复“我们必须敏捷”来掩盖组织中的根本问题。

# 我的全堆叠经验。

我在一家不关心前端和后端的公司工作。每个人都是“开发人员”，我的角色是 PHP、MySQL、JS、CSS、HTML 的混合体，任何需要的东西都有。我们没有任何大的 web 项目要构建——现有 CMS 中的一些模板系统、邮件功能、与不同工具和服务的集成。这些项目更像是小组织或初创公司的项目。十多年前，前端开发更加简单。几年后，我们将团队分为后端和前端。

我在另一个似乎支持全栈方法的地方有经验，我看不出它如何解决任何问题。人们最终还是无所事事；这并不能省钱，因为没有真正的责任和所有权，人们更不专注，效率更低。我很肯定这也转化为团队成员的快乐，因为他们不想成为一个人的军刀(至少他们中的一些人)。

与全堆栈形成对比的是，在另一家公司，我们接受了拆分，其中 95%的工程团队是半堆栈，5%是全堆栈开发人员。这种方法非常有效。我们建立了混合的 sprint 团队，他们紧密合作，谈论很多他们的东西，分享他们的经验和最佳实践。团队开发了工具和方法，帮助他们专注于他们的专业领域，以一种在过程中不会互相阻碍的方式交付产品。我们建立信任和关系来提供速度和质量。在紧急情况下，我们是一个团队，互相支持，互相帮助。

> “如果你想走得快，就一个人走。想走远，就一起走。”

我有更多的例子表明，专业化无论如何都不是一个障碍，你不需要明确地雇佣“全栈开发人员”来填补空白以保持灵活性。你需要合适类型的人和合适的职业环境。这才是真正重要的。

# 前端=前端，后端=全栈？

旁注…我注意到一个模式。前端开发人员似乎理解并大多同意这种分离。他们知道现在前端有多庞大，要跟上时代有多难。他们还意识到，要做好事情，后端也需要同样多的知识。你的人生没有足够的时间去超越一切，所以最好至少做对一件事。

后端开发人员通常认为自己是全栈的。当我在工作中，在论坛上，在 LinkedIn 上问我的同事时，我注意到了这一点。有例外，但每条规则都有例外。所以问题是——为什么？他们以为前端就这么简单；下载几个模块，放在一起，就搞定了？他们做了很多前端工作，因为他们不能只发布后端？后端编码很容易，所以他们觉得前端也很容易？我不知道，这只是一个理论，但我觉得它非常有趣，耐人寻味。
