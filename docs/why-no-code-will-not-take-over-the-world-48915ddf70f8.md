## 2021 年 11 月，我建立了我的第一个无代码项目。

太神奇了。

我从想法到产品的速度就像施了一个咒语。

如果海格站在我面前说“你是个巫师，马塞尔”，我不会感到惊讶。

我理解所有那些说**“无代码是未来”**“软件工程的时代已经一去不复返了”的推文。

我被吸引住了，我开始做自由职业者，成为一名无代码开发人员，我变得很棒。我找到了一个联合创始人，我们开始了一个机构。

我们开始大力投资无代码。为初创企业、风险投资公司和所有处于这项新技术风口浪尖的人建立项目。

**但它不能永远待在罗西身边**。出事了。

Bubble.io(我们选择的无代码工具)宣布他们正在改变他们的定价结构。

现在，价格变化实际上不会影响我们(因为他们对代理商有不同的定价)，但**这是一个危险信号**。

有点像你约会的人对服务员很粗鲁——但他们通常对你很好。这是一个我们不能忽视的危险信号。

我开始认真考虑是否还有其他我遗漏的问题。

**我考虑了使用无代码构建的所有限制**和问题，我想**与你分享我的发现**。

在接下来的内容之前，我会说我仍然相信这项技术，我们的机构仍然接受无代码项目，但我们清楚这些限制，并尽可能保护我们的客户免受这些限制。

# **定价地毯拉动**

以无代码为代码基础创建一家初创企业，就像开了一家酒店，却不拥有这栋楼。

但更糟糕的是，你不仅不拥有这栋楼，你也没有与业主签订合同。你没有租约，他们没有合同义务，**他们可以随时提高租金。**

也许他们会增加工资，因为他们看到你做得很好。

也许他们会增加投资，因为他们的投资者想要一个好的退出机会。谁知道呢？

原因并不重要，但你是在赌你的底线，希望他们会决定保持低价，并且他们也会管理好自己的业务，以保持平台的运行。

Bubble 的[价格上涨](https://forum.bubble.io/t/changes-to-how-we-charge-for-applications-going-forward/196386)实际上并不是价格的直接上涨，而是定价结构的转变，新价格主要基于您的数据库需求的大小。

这不可避免地意味着，如果你使用该平台构建的产品取得了成功，你的成本可能会增加。

社区的反弹是巨大的，足以让 Bubble 彻底改变这一变化，但引起我注意的是，在评论的深处，你可以看到人们在嘀咕 Bubble 早期的一个类似事件。

泡泡在之前也做过类似[的事情。](https://forum.bubble.io/t/pricing-updates-and-thoughts/64515)

![](img/f13d6f63cdb7cfe64d3905c9d7d5967d.png)

这一次的价格变化是基于数据库大小的，这一事实尤其令人担忧。

这意味着随着你的成长，你如何确定**定义你成功的东西**“你有越多的交互(你收集的数据)”不会成为**一个无代码平台惩罚你的东西。**

# **你没有自己的代码**

继续酒店的类比，你仍然不拥有这栋楼。

如果，由于某种原因，拥有这栋楼的公司倒闭了，没有合同说下一个业主会继续你的安排。

更具体地说，**如果无代码平台破产，那么你的代码也会破产。**

Bubble 已经做出保证，如果发生这种情况，他们将开放他们的代码，这是一个很好的保证，但事实是，如果发生这种情况，那么你的产品就泡汤了。没有技术的支持和改进，它可能会慢慢衰退。

对于想要创造 MVP 的普通企业家来说，这不是一个问题，但是对于想要创造新产品的财富 500 强公司来说，这是一个严重的限制。

> *如果你经营一家科技企业，但没有自己的代码，你是一家科技企业吗？*

# **不可避免的限制**

我们已经使用无代码构建了许多不同类型的项目，我们已经意识到，你的项目不可避免地会遇到无代码无法解决的限制。

大多数企业都乐于解决这些问题，特别是如果他们只是为了获得最初的吸引力而进行建设，但在某些时候这些问题会成为最重要的问题。我们观察到的这种**限制包括:**

*   **速度**。构建一个无代码的数据库意味着你没有能力索引你的数据库，建立主键和其他一些东西。问题的关键在于，当您成功时，您的数据库大小可能会降低应用程序的性能
*   **无法在移动应用上构建原生功能**。利用手机推送通知、谷歌或苹果支付、手机媒体播放器等功能。没有代码是不可能的
*   **离线访问**。如果有人在一个信号差/没有信号的地区，应用程序将无法使用，而不是建立一个本地应用程序
*   **数据操作** —无代码平台在运行逻辑上做得不好。这可能简单到根据嵌套的 if 语句显示信息，也可能复杂到运行一个算法来查找某人的日程表。两者都不能很好地使用无代码

如果你计划得好，你或你的团队会意识到你将面临的限制。

人们通常对此感到高兴，因为他们认为“如果我们到了这是一个问题的地步，那就意味着我们无论如何都处于一个有利的位置！**明天的我可以应付**。老实说，这很有道理。我们经常意见一致。

但是**在某些时候，明天的你必须面对它。**您的客户一次又一次地要求该功能，您意识到它已经成为一种必需品，而不是一种很好拥有的功能。

这时你意识到你需要从无代码转换到传统代码，但是你有一个问题。技术债。

# **技术债务**

“技术债务，描述了当开发团队采取行动来加速一项功能或一个项目的交付时会产生什么结果，这些功能或项目随后需要进行重构。”如[产品计划所定义。](https://www.productplan.com/glossary/technical-debt/)

本质上，快速构建软件和快速构建一个建筑没有什么不同。想象一下，你正在漫不经心地建造一座叠人塔。在某种程度上，它要么下降，要么你意识到你不能继续添加更多的层(在这种情况下的功能)而不危及整个事情。进展缓慢，直到事情被重构。

多年来对传统代码的改进，意味着你可以小心翼翼地构建传统代码，并且做得足够好，以至于这个时刻可能永远不会到来，或者当它到来时，你的代码库足够干净，你可以在没有太多问题的情况下进行重构。

对于无代码，这可能意味着你真的需要重新开始。你有一个天花板，没有办法超越它。要么是因为代码库无法处理大量的改进，要么是因为无代码不够灵活，无法添加新功能。

你使用无代码的事实将会减缓改进的速度。

其原因本质上是技术性的，但除了上述限制之外，还包括:

*   **分支、版本控制和协作是痛苦的。Git 彻底改变了开发。而一个很好地使用版本控制的开发团队是一道美丽的风景。Bubble 有自己的版本控制，但比 Git 落后好几光年。**
*   **构建结构化的模块化代码很难。**很难构建孤立的代码，这意味着孤立的特性可能很难重构
*   **国家管理困难。你必须在页面之间或者在页面 url 中传递变量。这实质上意味着复杂的应用程序变得越大越混乱**
*   **你不能自动化测试。**如果你不能添加测试，随着事情变大，你必须手动确保你的新特性没有破坏其他东西。

这个清单还可以继续下去，但关键是，为了换取速度，你放弃了多年来在软件开发方面的技术创新。

当你达到你的上限时，你可能可以在没有改善的情况下经营你的企业，或者在某个时候你需要从头开始“还债”。

# **这对使用无代码开发意味着什么**

最终，无代码解决了一个非常非常现实的问题。如果你的项目可以基于 web 应用，那么你获得 MVP 的速度将是不可思议的。但你必须意识到，如果/当你获得成功，你可能会遇到限制，会有技术债务和建立在你不拥有的土地上的基础。

# **这对我们的发展方式意味着什么**

无代码仍然是我们储物柜中的一个工具，但我们已经增加了对 Flutter 和其他通用编码技术的投资和雇佣。

虽然有时我们可以让客户从无代码开始，但他们不可避免地会转向其他技术，而且通常其他技术可能从一开始就更合适。

如果有人对自己的牵引力有信心，并且有资金来构建健壮且可伸缩的东西，那么无代码就没有用武之地。

如果在确定项目范围后，我们发现无代码是最合适的工具，我们从一开始就设定期望。也就是说，你会很快得到一些东西，这将有助于你获得牵引力，但你可能需要在以后切换。

如果客户想要无代码，但有更多的预算，我们将建立一个外部数据库的无代码前端。这意味着我们使用 noSQL 或 SQL 构建数据库，然后使用 API 在无代码前端显示数据。这意味着如果我们需要迁移到另一个工具，颤动，反应等等。然后，我们可以构建新的前端并连接到外部数据库。这减轻了迁移的影响，当我们的客户遇到这种情况时，更容易将应用程序带到下一个级别。

感谢你阅读一个前无代码传道者的漫谈。或者可能是对正常代码的新信徒。对于我的联合创始人桑尼，6 个月后，我很高兴地说，你是对的。无代码不会接管世界，至少暂时不会。
