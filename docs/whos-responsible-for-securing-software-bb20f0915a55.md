# 谁负责保护软件？

> 原文：<https://medium.com/geekculture/whos-responsible-for-securing-software-bb20f0915a55?source=collection_archive---------57----------------------->

在一个拥抱 DevOps 的世界里，Infosec 不确定哪一个应该承担这个责任。

![](img/ea7a04d9cc1ce5a773fb76a6b11c47a0.png)

Photo by [maxxyustas](https://elements.envato.com/user/maxxyustas) from [Envato Elements](https://elements.envato.com/computer-security-or-safety-concept-laptop-keyboar-PQ7RZZ5)

一份 [Venify 报告](https://www.venafi.com/resource/more-solarwinds-style-attacks-whitepaper)显示，由 DevOps 负责安全的公司数量和将其分配给信息安全部门的公司数量几乎各半。但是，关于软件安全的责任，真的应该有这种模糊性吗？

当你看到纸上的情况时，你可能会说信息安全应该在应用程序中找到漏洞，因为它们通常会对公司的运行软件进行渗透测试。但同样令人信服的是，由于 DevOps 知道软件如何工作，并且可能是少数几个可以访问代码库的单位之一，他们应该确保黑客无法利用该软件。

从这个角度看，似乎这两种观点都是对的。开发人员应该修补他们的程序——并确保没有那些讨厌的一个接一个的错误——而信息安全部门拥有必要的知识来使正在运行的程序脱离正轨。

因此，对你和我来说，这两个部门应该扮演什么角色似乎是显而易见的。但是当网络攻击发生时，谁来承担责任呢？

我不是提出这个问题的人。如果 DevOps 或 Infosec 都存在的话，管理人员通常会将这一职责分配给它们。我们没有看到大多数企业在被黑时分担集体责任，[对吗？](https://www.theverge.com/2017/10/3/16410806/equifax-ceo-blame-breach-patch-congress-testimony)那么，他们为什么不设立一个部门来负责保护公司免受攻击呢？

这不是解决问题的好方法，因为仅仅测试程序的漏洞忽略了一个事实，即黑客可以尝试一种全新的攻击手段来进入组织。虽然 DevOps 可以防范一些基本缺陷，如空解引用，但他们没有时间让 Infosec 运行不同的测试来不断捕捉新的漏洞。

我不认为事件响应应该是一个相互指责的游戏，在应用程序交互过程中发挥作用的所有部门都应该负责保护他们在公司使用的软件，不仅为他们自己的公司，也为在他们的产品中使用该软件的客户。
